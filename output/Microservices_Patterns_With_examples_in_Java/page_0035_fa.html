The slow march toward monolithic hell

برای بدتر کردن اوضاع، این پیچیدگی طاقت‌فرسا تمایل به یک مارپیچ نزولی دارد. اگر code base دشوار است، یک developer تغییرات را به درستی انجام نمی‌دهد. هر تغییر باعث می‌شود code base به طور فزاینده‌ای پیچیده‌تر و درک آن دشوارتر شود. معماری تمیز و ماژولار که قبلاً در شکل 1.1 نشان داده شد، واقعیت را منعکس نمی‌کند. FTGO به تدریج در حال تبدیل شدن به یک توپ بزرگ و غیرقابل درک و هیولایی از گل است.

ماری به یاد می‌آورد که اخیراً در کنفرانسی شرکت کرده بود که در آن با developerی ملاقات کرد که در حال نوشتن ابزاری برای تجزیه و تحلیل وابستگی‌ها بین هزاران JAR در application چند میلیون خطی (LOC) بود. در آن زمان، آن ابزار چیزی به نظر می‌رسید که FTGO می‌توانست از آن استفاده کند. اکنون او مطمئن نیست. ماری گمان می‌کند که یک رویکرد بهتر، مهاجرت به معماری است که برای یک application پیچیده مناسب‌تر است: microservices.

DEVELOPMENT IS SLOW

علاوه بر مبارزه با پیچیدگی طاقت‌فرسا، developers FTGO وظایف توسعه روزانه را کند می‌یابند. application بزرگ، IDE developer را اضافه بار می‌کند و کند می‌کند. ساخت application FTGO زمان زیادی می‌برد. علاوه بر این، از آنجایی که بسیار بزرگ است، راه‌اندازی application زمان زیادی می‌برد. در نتیجه، حلقه ویرایش-ساخت-اجرا-تست زمان زیادی می‌برد که به شدت بر بهره‌وری تأثیر می‌گذارد.

PATH FROM COMMIT TO DEPLOYMENT IS LONG AND ARDUOUS
