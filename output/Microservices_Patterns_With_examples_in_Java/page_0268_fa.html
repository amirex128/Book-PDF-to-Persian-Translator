CHAPTER 7

Implementing queries in a microservice architecture

NoSQL databases. یک database NoSQL معمولاً دارای یک فرم محدود از transactions و قابلیت‌های querying کمتر عمومی است. با این حال، برای موارد استفاده خاص، این databases دارای مزایای خاصی نسبت به SQL databases هستند، از جمله یک مدل data انعطاف‌پذیرتر و عملکرد و مقیاس‌پذیری بهتر.

یک database NoSQL اغلب یک انتخاب خوب برای یک CQRS view است، که می‌تواند از نقاط قوت آن استفاده کند و نقاط ضعف آن را نادیده بگیرد. یک CQRS view از مدل data غنی‌تر و عملکرد یک database NoSQL بهره می‌برد. این تحت تأثیر محدودیت‌های یک database NoSQL قرار نمی‌گیرد، زیرا فقط از transactions ساده استفاده می‌کند و یک مجموعه ثابت از queries را اجرا می‌کند.

با این حال، گاهی اوقات منطقی است که یک CQRS view را با استفاده از یک SQL database پیاده‌سازی کنید. یک RDBMS مدرن که روی سخت‌افزار مدرن اجرا می‌شود، عملکرد بسیار خوبی دارد. توسعه‌دهندگان، مدیران database و عملیات IT، به‌طور کلی، با SQL databases بسیار بیشتر از databases NoSQL آشنا هستند. همانطور که قبلاً ذکر شد، SQL databases اغلب دارای افزونه‌هایی برای ویژگی‌های غیر رابطه‌ای، مانند انواع داده‌های فضایی و queries هستند. همچنین، یک CQRS view ممکن است برای پشتیبانی از یک موتور گزارش‌گیری، نیاز به استفاده از یک SQL database داشته باشد.

همانطور که در جدول 7.1 مشاهده می‌کنید، گزینه‌های زیادی برای انتخاب وجود دارد. و برای پیچیده‌تر کردن انتخاب، تفاوت بین انواع مختلف database شروع به محو شدن می‌کند. به عنوان مثال، MySQL، که یک RDBMS است، از JSON پشتیبانی عالی دارد، که یکی از نقاط قوت MongoDB است، یک database با قالب سند JSON.

اکنون که در مورد انواع مختلف databases که می‌توانید برای پیاده‌سازی یک CQRS view استفاده کنید، بحث کردم، بیایید به مشکل نحوه به‌روزرسانی کارآمد یک view نگاهی بیندازیم.
