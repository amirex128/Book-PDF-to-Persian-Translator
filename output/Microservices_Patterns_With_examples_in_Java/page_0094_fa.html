فصل 2

Decomposition strategies

داده‌ها در services متعدد، یک saga نیز راهی برای پیاده‌سازی یک service خود-محصور است. به عنوان مثال، من شرح می‌دهم که چگونه عملیات createOrder() با استفاده از یک saga پیاده‌سازی می‌شود، که servicesی مانند Consumer Service، Kitchen Service و Accounting Service را با استفاده از پیام‌رسانی ناهمزمان فراخوانی می‌کند.

فصل 8 مفهوم یک API gateway را شرح می‌دهد که یک API را در معرض دید clients خارجی قرار می‌دهد. یک API gateway ممکن است یک عملیات query را با استفاده از الگوی ترکیب API، که در فصل 7 توضیح داده شد، پیاده‌سازی کند، نه اینکه صرفاً آن را به service هدایت کند. منطق موجود در API gateway، داده‌های مورد نیاز توسط query را با فراخوانی services متعدد و ترکیب نتایج جمع‌آوری می‌کند. در این موقعیت، عملیات سیستم به API gateway اختصاص داده می‌شود نه به یک service. Services باید عملیات query مورد نیاز توسط API gateway را پیاده‌سازی کنند.

Summary

  • معماری، -ilities application شما، از جمله قابلیت نگهداری، قابلیت تست، و قابلیت استقرار را تعیین می‌کند که مستقیماً بر سرعت توسعه تأثیر می‌گذارد.
  • معماری microservice یک سبک معماری است که به یک application قابلیت نگهداری، قابلیت تست و قابلیت استقرار بالایی می‌دهد.
  • Services در یک معماری microservice حول نگرانی‌های کسب‌وکار - قابلیت‌های کسب‌وکار یا زیر domains - و نه نگرانی‌های فنی سازماندهی می‌شوند.
  • دو الگو برای تجزیه وجود دارد:
    • - تجزیه بر اساس قابلیت کسب‌وکار، که ریشه در معماری کسب‌وکار دارد
    • - تجزیه بر اساس زیر domain، بر اساس مفاهیم طراحی مبتنی بر domain
  • شما می‌توانید کلاس‌های god را حذف کنید، که باعث وابستگی‌های درهم می‌شوند که از تجزیه جلوگیری می‌کنند، با اعمال DDD و تعریف یک مدل domain جداگانه برای هر service.
