فرار از جهنم Monolithic

کتاب The Righteous Mind: Why Good People Are Divided by Politics and Religion (Vintage, 2013)، از استعاره یک فیل و سوارکارش برای توصیف چگونگی عملکرد ذهن انسان استفاده می‌کند. فیل نشان‌دهنده قسمت احساسی مغز انسان است. این فیل اکثر تصمیمات را می‌گیرد. سوارکار نشان‌دهنده بخش منطقی مغز است. این سوارکار گاهی اوقات می‌تواند بر فیل تأثیر بگذارد، اما بیشتر توجیهاتی را برای تصمیمات فیل ارائه می‌دهد.

ما - جامعه توسعه software - باید بر طبیعت احساسی خود غلبه کنیم و راه بهتری برای بحث و اعمال فناوری پیدا کنیم. یک راه عالی برای بحث و توصیف فناوری، استفاده از فرمت pattern است، زیرا عینی است. به عنوان مثال، هنگام توصیف یک فناوری در قالب pattern، باید معایب را توصیف کنید. بیایید نگاهی به قالب pattern بیندازیم.

1.6.2

Patterns and pattern languages

یک pattern یک راه‌حل قابل استفاده مجدد برای یک مشکل است که در یک زمینه خاص رخ می‌دهد. این ایده‌ای است که منشأ آن در معماری دنیای واقعی است و ثابت شده است که در معماری و طراحی software مفید است. مفهوم یک pattern توسط Christopher Alexander، یک معمار دنیای واقعی، ایجاد شد. او همچنین مفهوم یک زبان pattern، مجموعه‌ای از الگوهای مرتبط که مشکلات را در یک دامنه خاص حل می‌کنند، ایجاد کرد. کتاب او A Pattern Language: Towns, Buildings, Construction (Oxford University Press, 1977) یک زبان pattern برای معماری را توصیف می‌کند که از 253 pattern تشکیل شده است. الگوها از راه‌حل‌هایی برای مشکلات سطح بالا، مانند مکان قرارگیری یک شهر (”دسترسی به آب”)، تا مشکلات سطح پایین، مانند نحوه طراحی یک اتاق (”نور در دو طرف هر اتاق”) متغیر هستند. هر یک از این الگوها یک مشکل را با چیدمان اشیاء فیزیکی که دامنه آن‌ها از شهرها تا پنجره‌ها متغیر است، حل می‌کند.

نوشته‌های Christopher Alexander، جامعه software را برانگیخت تا مفهوم الگوها و زبان‌های pattern را اتخاذ کنند. کتاب Design Patterns: Elements of Reusable Object-Oriented Software (Addison-Wesley Professional, 1994)، توسط Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides مجموعه‌ای از الگوهای طراحی شی گرا است. این کتاب الگوها را در بین developers software محبوب کرد. از اواسط دهه 1990، developers software الگوهای software متعددی را مستند کرده‌اند. یک pattern software، یک معماری یا مشکل طراحی software را با تعریف مجموعه‌ای از عناصر software در حال همکاری حل می‌کند.

به عنوان مثال، بیایید تصور کنیم که شما در حال ساخت یک application بانکی هستید که باید از انواع سیاست‌های overdraft پشتیبانی کند. هر سیاست محدودیت‌هایی را در مورد موجودی حساب و هزینه‌های دریافتی برای یک حساب overdrawn تعریف می‌کند. شما می‌توانید این مشکل را با استفاده از الگوی Strategy، که یک pattern شناخته شده از کتاب کلاسیک Design Patterns است، حل کنید. راه‌حل تعریف شده توسط الگوی Strategy از سه بخش تشکیل شده است:

  • یک رابط استراتژی به نام Overdraft که الگوریتم overdraft را کپسوله می‌کند
  • یک یا چند کلاس استراتژی بتنی، یکی برای هر زمینه خاص
  • کلاس Account که از الگوریتم استفاده می‌کند

الگوی Strategy یک الگوی طراحی شی گرا است، بنابراین عناصر راه‌حل کلاس هستند. در ادامه این بخش، الگوهای طراحی سطح بالا را شرح می‌دهم، که در آن راه‌حل از services در حال همکاری تشکیل شده است.
