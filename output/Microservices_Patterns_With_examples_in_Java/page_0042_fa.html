1.4.3

Each service has its own database

یک ویژگی کلیدی معماری microservice این است که services loosely coupled هستند و فقط از طریق APIs ارتباط برقرار می‌کنند. یک راه برای دستیابی به loose coupling این است که هر service datastore خود را داشته باشد. به عنوان مثال، در فروشگاه آنلاین، Order Service دارای یک پایگاه داده است که شامل جدول ORDERS است، و Customer Service دارای پایگاه داده خود است که شامل جدول CUSTOMERS است. در زمان توسعه، developers می‌توانند طرحواره یک service را بدون نیاز به هماهنگی با developers که روی services دیگر کار می‌کنند، تغییر دهند. در زمان اجرا، services از یکدیگر جدا شده‌اند - به عنوان مثال، یک service هرگز مسدود نخواهد شد زیرا service دیگری قفل پایگاه داده را نگه می‌دارد.

اکنون که معماری microservice را تعریف کردیم و برخی از ویژگی‌های ضروری آن را توصیف کردیم، بیایید نگاهی بیندازیم که چگونه این امر برای application FTGO اعمال می‌شود.

1.4.4

The FTGO microservice architecture

بقیه این کتاب، معماری microservice application FTGO را با جزئیات مورد بحث قرار می‌دهد. اما ابتدا بیایید به سرعت نگاهی بیندازیم که اعمال مقیاس‌بندی محور Y به این application چه معنایی دارد. اگر تجزیه محور Y را روی application FTGO اعمال کنیم، معماری نشان داده شده در شکل 1.7 را به دست می‌آوریم. application تجزیه شده از services frontend و backend متعددی تشکیل شده است. ما همچنین مقیاس‌بندی محور X و احتمالاً Z را اعمال می‌کنیم، به طوری که در زمان اجرا چندین نمونه از هر service وجود داشته باشد.

services frontend شامل یک API gateway و Restaurant Web UI است. API gateway، که نقش facade را ایفا می‌کند و در فصل 8 با جزئیات توضیح داده شده است، REST APIs را ارائه می‌دهد که توسط applications mobile مصرف‌کنندگان و پیک‌ها استفاده می‌شود. Restaurant Web UI رابط وب را پیاده‌سازی می‌کند که توسط رستوران‌ها برای مدیریت منوها و پردازش سفارشات استفاده می‌شود.

business logic application FTGO از services backend متعددی تشکیل شده است. هر service backend دارای یک REST API و datastore خصوصی خود است. services backend شامل موارد زیر است:

  • Order Service—مدیریت سفارشات
  • Delivery Service—مدیریت تحویل سفارشات از رستوران‌ها به مصرف‌کنندگان
