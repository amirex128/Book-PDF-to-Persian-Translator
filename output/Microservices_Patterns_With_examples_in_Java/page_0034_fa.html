فرار از جهنم Monolithic

از معماری software، که یک سیستم را به عنوان یک مؤلفه قابل اجرا یا استقرار واحد ساختار می‌دهد. اگر application FTGO به زبان Go (GoLang) نوشته شده بود، یک فایل اجرایی واحد بود. نسخه Ruby یا NodeJS از این application یک سلسله‌مراتب دایرکتوری از کد منبع خواهد بود. معماری monolithic ذاتاً بد نیست. developers FTGO زمانی که معماری monolithic را برای application خود انتخاب کردند، تصمیم درستی گرفتند.

1.1.2

The benefits of the monolithic architecture

در روزهای اولیه FTGO، زمانی که application نسبتاً کوچک بود، معماری monolithic application مزایای زیادی داشت:

  • ساختن Simple — IDEs و سایر ابزارهای developer بر ساختن یک application واحد متمرکز هستند.
  • آسان برای ایجاد تغییرات اساسی در application - شما می‌توانید کد و database schema را تغییر دهید، بسازید و استقرار دهید.
  • تست کردن Straightforward — developers تست‌های end-to-end را نوشتند که application را راه‌اندازی می‌کرد، REST API را فراخوانی می‌کرد و UI را با Selenium تست می‌کرد.
  • راه‌اندازی Straightforward — تنها کاری که یک developer باید انجام می‌داد این بود که فایل WAR را در یک سروری که Tomcat نصب شده بود کپی کند.
  • Easy to scale—FTGO چندین نمونه از application را در پشت یک load balancer اجرا کرد.

با گذشت زمان، توسعه، تست، استقرار و مقیاس‌بندی بسیار دشوارتر شد. بیایید نگاهی بیندازیم که چرا.

1.1.3

Living in monolithic hell

متأسفانه، همانطور که developers FTGO کشف کرده‌اند، معماری monolithic دارای یک محدودیت بزرگ است. applications موفق مانند application FTGO عادت دارند که از معماری monolithic فراتر بروند. در هر sprint، تیم توسعه FTGO چند داستان دیگر را پیاده‌سازی کرد که باعث بزرگتر شدن code base شد. علاوه بر این، با موفقیت بیشتر شرکت، اندازه تیم توسعه به طور پیوسته افزایش یافت.

این نه تنها باعث افزایش سرعت رشد code base شد، بلکه سربار مدیریت را نیز افزایش داد.

همانطور که شکل 1.2 نشان می‌دهد، application FTGO که زمانی کوچک و ساده بود، در طول سال‌ها به یک monolith هیولا تبدیل شده است. به طور مشابه، تیم توسعه کوچک اکنون به تیم‌های Scrum متعددی تبدیل شده است که هر کدام بر روی یک حوزه عملکردی خاص کار می‌کنند. در نتیجه فراتر رفتن از معماری خود، FTGO در جهنم monolithic قرار دارد. توسعه کند و دردناک است. توسعه و استقرار Agile غیرممکن است. بیایید نگاهی بیندازیم که چرا این اتفاق افتاده است.

COMPLEXITY INTIMIDATES DEVELOPERS

یک مشکل اساسی در application FTGO این است که بیش از حد پیچیده است. برای اینکه هر developer آن را به طور کامل درک کند، بسیار بزرگ است. در نتیجه، رفع اشکالات و پیاده‌سازی صحیح ویژگی‌های جدید دشوار و زمان‌بر شده است. مهلت‌ها از دست می‌روند.
