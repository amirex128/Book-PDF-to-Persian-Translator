CHAPTER 7

Implementing queries in a microservice architecture

با اشتراک در events منتشر شده توسط چندین services. این نوع view به هیچ service خاصی تعلق ندارد، بنابراین منطقی است که آن را به عنوان یک service مستقل پیاده‌سازی کنید. یک مثال خوب از این service، Order History Service است، که یک query service است که عملیات query findOrderHistory() را پیاده‌سازی می‌کند. همانطور که شکل 7.9 نشان می‌دهد، این service به events منتشر شده توسط چندین services، از جمله Order Service، Delivery Service و غیره، مشترک می‌شود.

Order History Service دارای event handlers است که به events منتشر شده توسط چندین services مشترک می‌شوند و Order History View Database را به‌روزرسانی می‌کنند. من پیاده‌سازی این service را با جزئیات بیشتر در بخش 7.4 توضیح می‌دهم.

یک query service همچنین یک راه خوب برای پیاده‌سازی یک view است که داده‌های متعلق به یک service واحد را تکرار می‌کند، اما به دلیل نیاز به جداسازی نگرانی‌ها، بخشی از آن service نیست.
به عنوان مثال، توسعه‌دهندگان FTGO می‌توانند یک Available Restaurants Service را تعریف کنند، که عملیات query findAvailableRestaurants() را که قبلاً توضیح داده شد، پیاده‌سازی می‌کند. این سرویس به events منتشر شده توسط Restaurant Service مشترک می‌شود و یک database را که برای queries فضایی کارآمد طراحی شده است، به‌روزرسانی می‌کند.

از بسیاری جهات، CQRS یک تعمیم مبتنی بر event از رویکرد محبوب استفاده از RDBMS به عنوان سیستم سوابق و یک موتور جستجوی متنی، مانند Elasticsearch، برای رسیدگی به text queries است. آنچه متفاوت است این است که CQRS از طیف وسیع‌تری از database
