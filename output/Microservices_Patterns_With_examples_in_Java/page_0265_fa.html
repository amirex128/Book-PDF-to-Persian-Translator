7.2.3 The benefits of CQRS

CQRS هم مزایا و هم معایبی دارد. مزایا به شرح زیر است:

  • امکان پیاده‌سازی کارآمد queries در یک معماری microservice
  • امکان پیاده‌سازی کارآمد queries متنوع
  • امکان querying را در یک application مبتنی بر event sourcing فراهم می‌کند
  • جداسازی نگرانی‌ها را بهبود می‌بخشد
ENABLES THE EFFICIENT IMPLEMENTATION OF QUERIES IN A MICROSERVICE ARCHITECTURE

یکی از مزایای الگوی CQRS این است که queriesی را که داده‌ها را از چندین services بازیابی می‌کنند، به طور کارآمد پیاده‌سازی می‌کند. همانطور که قبلاً توضیح داده شد، استفاده از الگوی ترکیب API برای پیاده‌سازی queries گاهی اوقات منجر به joins درون حافظه‌ای پرهزینه و ناکارآمد از datasets بزرگ می‌شود. برای آن queries، استفاده از یک CQRS view که به راحتی query می‌شود و داده‌ها را از دو یا چند services از قبل join می‌کند، کارآمدتر است.

ENABLES THE EFFICIENT IMPLEMENTATION OF DIVERSE QUERIES

یکی دیگر از مزایای CQRS این است که یک application یا service را قادر می‌سازد تا مجموعه متنوعی از queries را به طور کارآمد پیاده‌سازی کند. تلاش برای پشتیبانی از همه queries با استفاده از یک مدل dataی پایدار واحد اغلب چالش‌برانگیز و در برخی موارد غیرممکن است. برخی از databases NoSQL قابلیت‌های querying بسیار محدودی دارند. حتی زمانی که یک database دارای افزونه‌هایی برای پشتیبانی از یک نوع خاص از query است، استفاده از یک database تخصصی اغلب کارآمدتر است. الگوی CQRS با تعریف یک یا چند views، که هر کدام به طور کارآمد queries خاصی را پیاده‌سازی می‌کنند، از محدودیت‌های یک datastore واحد جلوگیری می‌کند.

ENABLES QUERYING IN AN EVENT SOURCING-BASED APPLICATION

CQRS همچنین بر یک محدودیت اصلی event sourcing غلبه می‌کند. یک event store فقط از queries مبتنی بر primary key پشتیبانی می‌کند. الگوی CQRS با تعریف یک یا چند views از aggregates، که با اشتراک در جریان eventsی که توسط aggregates مبتنی بر event sourcing منتشر می‌شوند، به‌روز نگه داشته می‌شوند، به این محدودیت رسیدگی می‌کند. در نتیجه، یک application مبتنی بر event sourcing به‌طور قطع از CQRS استفاده می‌کند.

IMPROVES SEPARATION OF CONCERNS

یکی دیگر از مزایای CQRS این است که نگرانی‌ها را جدا می‌کند. یک مدل domain و مدل dataی پایدار مربوطه آن، هم دستورات و هم queries را مدیریت نمی‌کنند. الگوی CQRS ماژول‌های کد و schemas database جداگانه‌ای را برای سمت دستور و سمت query یک service تعریف می‌کند. با جداسازی نگرانی‌ها، سمت دستور و سمت query به احتمال زیاد ساده‌تر و نگهداری آن‌ها آسان‌تر است.

علاوه بر این، CQRS service را قادر می‌سازد که یک query را پیاده‌سازی کند و با serviceی که مالک data است، متفاوت باشد. به عنوان مثال، قبلاً توضیح دادم که چگونه، اگرچه Restaurant Service مالک data است که توسط عملیات query findAvailableRestaurants query می‌شود، منطقی است که یک service دیگر چنین
