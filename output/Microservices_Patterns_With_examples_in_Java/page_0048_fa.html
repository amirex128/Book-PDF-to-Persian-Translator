فرار از جهنم Monolithic

IDEs و سایر ابزارهای توسعه بر ساخت applications monolithic متمرکز شده‌اند و پشتیبانی صریحی برای توسعه applications توزیع شده ارائه نمی‌دهند. نوشتن تست‌های خودکار که شامل services متعدد می‌شود، یک چالش است. این‌ها همه مسائلی هستند که مختص معماری microservice هستند. در نتیجه، developers سازمان شما باید مهارت‌های پیچیده توسعه و تحویل software را داشته باشند تا بتوانند با موفقیت از microservices استفاده کنند.

معماری microservice همچنین پیچیدگی عملیاتی قابل توجهی را معرفی می‌کند. قطعات متحرک بسیار بیشتری - نمونه‌های متعددی از انواع مختلف service - باید در تولید مدیریت شوند. برای استقرار موفقیت‌آمیز microservices، به سطح بالایی از اتوماسیون نیاز دارید. شما باید از فناوری‌هایی مانند موارد زیر استفاده کنید:

  • ابزارهای استقرار خودکار، مانند Netflix Spinnaker
  • یک PaaS آماده، مانند Pivotal Cloud Foundry یا Red Hat OpenShift
  • یک پلتفرم Docker orchestration، مانند Docker Swarm یا Kubernetes

گزینه‌های استقرار را با جزئیات بیشتری در فصل 12 شرح می‌دهم.

DEPLOYING FEATURES SPANNING MULTIPLE SERVICES NEEDS CAREFUL COORDINATION

چالش دیگر در استفاده از معماری microservice این است که استقرار ویژگی‌هایی که چندین service را در بر می‌گیرند، به هماهنگی دقیق بین تیم‌های توسعه مختلف نیاز دارد. شما باید یک برنامه راه‌اندازی ایجاد کنید که استقرار service را بر اساس وابستگی‌های بین services سفارش دهد. این کاملاً متفاوت از معماری monolithic است، جایی که می‌توانید به راحتی به‌روزرسانی‌ها را به مؤلفه‌های متعدد به‌طور اتمی مستقر کنید.

DECIDING WHEN TO ADOPT IS DIFFICULT

مسئله دیگر در استفاده از معماری microservice این است که در چه مرحله‌ای از چرخه عمر application باید از این معماری استفاده کنید. هنگام توسعه اولین نسخه از یک application، اغلب مشکلاتی را که این معماری حل می‌کند، ندارید. علاوه بر این، استفاده از یک معماری پیچیده و توزیع شده، توسعه را کند می‌کند. این می‌تواند یک معضل بزرگ برای استارتاپ‌ها باشد، جایی که بزرگترین مشکل معمولاً این است که چگونه مدل کسب‌وکار و application همراه را به سرعت تکامل دهیم.

استفاده از معماری microservice، تکرار سریع را بسیار دشوارتر می‌کند. یک استارتاپ تقریباً مطمئناً باید با یک application monolithic شروع کند.

اما بعداً، زمانی که مشکل چگونگی رسیدگی به پیچیدگی است، در این زمان منطقی است که application را از نظر عملکردی به مجموعه‌ای از microservices تجزیه کنید. ممکن است refactoring را به دلیل وابستگی‌های درهم دشوار بدانید. فصل 13 استراتژی‌هایی را برای refactoring یک application monolithic به microservices بررسی می‌کند.

همانطور که می‌بینید، معماری microservice مزایای زیادی را ارائه می‌دهد، اما دارای معایب قابل توجهی نیز است. به دلیل این مسائل، اتخاذ معماری microservice نباید با بی‌توجهی انجام شود. اما برای applications پیچیده، مانند یک application web رو به مصرف‌کننده یا application SaaS، معمولاً انتخاب درستی است. سایت‌های شناخته شده‌ای مانند eBay (www.slideshare.net/RandyShoup/the-ebay-architecture-striking-a-balance-between-site-stability-feature-velocity-performance-and-cost)، Amazon.com، Groupon و Gilt همگی از یک معماری monolithic به معماری microservice تکامل یافته‌اند.
