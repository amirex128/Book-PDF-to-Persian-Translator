فرار از جهنم Monolithic

گاهی اوقات می‌توانید از الگوی API composition استفاده کنید، که APIs یک یا چند service را فراخوانی کرده و نتایج را جمع‌آوری می‌کند. زمان‌های دیگر، شما باید از الگوی تفکیک مسئولیت پرس و جوی دستور (CQRS) استفاده کنید، که یک یا چند نسخه کپی‌شده قابل جستجوی داده‌ها را حفظ می‌کند. فصل 7 به راه‌های مختلف پیاده‌سازی queries می‌پردازد.

SERVICE DEPLOYMENT PATTERNS

استقرار یک application monolithic همیشه آسان نیست، اما از این جهت که یک application واحد برای استقرار وجود دارد، ساده است. شما باید چندین نمونه از application را در پشت یک load balancer اجرا کنید.

در مقایسه، استقرار یک application مبتنی بر microservices بسیار پیچیده‌تر است. ممکن است ده‌ها یا صدها service وجود داشته باشد که به زبان‌ها و فریم‌ورک‌های مختلف نوشته شده‌اند. قطعات متحرک بیشتری وجود دارد که باید مدیریت شوند. شکل 1.15 الگوهای استقرار را نشان می‌دهد.

راه سنتی و اغلب دستی استقرار applications در یک قالب بسته‌بندی خاص زبان، به عنوان مثال فایل‌های WAR، مقیاس‌پذیر نیست تا از معماری microservice پشتیبانی کند. شما به یک زیرساخت استقرار بسیار خودکار نیاز دارید. در حالت ایده‌آل، باید از یک پلتفرم استقرار استفاده کنید که یک UI ساده (خط فرمان یا GUI) را برای استقرار و مدیریت services خود به developer ارائه می‌دهد. پلتفرم استقرار معمولاً بر اساس ماشین‌های مجازی (VMs)، containers یا فناوری serverless خواهد بود. فصل 12 به گزینه‌های استقرار مختلف می‌پردازد.
