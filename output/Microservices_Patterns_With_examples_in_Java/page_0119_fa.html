Communicating using the Asynchronous messaging pattern

IMPLEMENTING ONE-WAY NOTIFICATIONS

پیاده‌سازی one-way notifications با استفاده از پیام‌رسانی asynchronous ساده است. کلاینت یک پیام، معمولاً یک پیام command، را به یک کانال نقطه به نقطه ارسال می‌کند متعلق به سرویس. سرویس در کانال مشترک می‌شود و mes- پیام را پردازش می‌کند. آن یک reply را برنمی‌گرداند.

IMPLEMENTING PUBLISH/SUBSCRIBE

پیام‌رسانی از تعامل سبک انتشار/اشتراک پشتیبانی داخلی دارد. یک کلاینت یک پیام را به یک کانال انتشار-اشتراک که توسط چندین مصرف‌کننده خوانده می‌شود، منتشر می‌کند. همانطور که در فصل‌های 4 و 5 توضیح داده شد، سرویس‌ها از انتشار/اشتراک برای انتشار استفاده می‌کنند domain events، که نشان‌دهنده تغییراتی در domain objects است. سرویسی که منتشر می‌کند domain events دارای یک کانال انتشار-اشتراک است که نام آن از کلاس domain. به عنوان مثال، Order Service، رویدادهای Order را به یک Order منتشر می‌کند کانال، و Delivery Service، رویدادهای Delivery را به یک Delivery chan- nel منتشر می‌کند. سرویسی که به رویدادهای یک domain object خاص علاقه‌مند است، فقط باید در کانال مناسب مشترک شوید.

IMPLEMENTING PUBLISH/ASYNC RESPONSES

سبک تعامل publish/async responses یک سبک تعامل سطح بالاتری است که با ترکیب عناصر انتشار/اشتراک و request/response پیاده‌سازی شده است. یک cli- یک کلاینت پیامی را منتشر می‌کند که یک هدر کانال reply را به یک انتشار-اشتراک مشخص می‌کند کانال. یک مصرف‌کننده، یک پیام reply حاوی یک correlation id را به reply می‌نویسد کانال. کلاینت، پاسخ‌ها را با استفاده از correlation id برای مطابقت با reply جمع‌آوری می‌کند پیام‌ها با request.

هر سرویس در اپلیکیشن شما که دارای یک API asynchronous است از یکی یا بیشتر از این تکنیک‌های پیاده‌سازی استفاده می‌کند. سرویسی که دارای یک API asynchronous برای فراخوانی عملیات، یک کانال پیام برای requests خواهد داشت. به طور مشابه، یک سرویس که رویدادها را منتشر می‌کند، آنها را به یک کانال پیام event منتشر می‌کند.

همانطور که در بخش 3.1.2 توضیح داده شد، نوشتن یک مشخصات API برای یک ser- vice، مهم است. بیایید نگاهی بیندازیم به نحوه انجام این کار برای یک API asynchronous.

3.3.3 Creating an API specification for a messaging-based service API

مشخصات API asynchronous یک سرویس، همانطور که شکل 3.9 نشان می‌دهد، باید مشخص کند نام‌های کانال‌های پیام، انواع پیام‌هایی که از طریق هر کدام رد و بدل می‌شوند کانال و فرمت‌های آنها. شما همچنین باید فرمت پیام‌ها را با استفاده از یک استاندارد مانند JSON، XML یا Protobuf. اما برخلاف REST و Open API، هیچ استاندارد گسترده‌ای برای مستندسازی کانال‌ها و انواع پیام‌ها وجود ندارد. در عوض، شما باید یک سند غیررسمی بنویسید.

API asynchronous یک سرویس شامل عملیات، که توسط کلاینت‌ها فراخوانی می‌شود و رویدادها، منتشر شده توسط سرویس‌ها. آنها به روش‌های مختلف مستند شده‌اند. بیایید نگاهی بیندازیم به هر کدام، با شروع از عملیات.
