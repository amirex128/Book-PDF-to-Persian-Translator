Decomposition strategies

آداپتور، business logic را فراخوانی می‌کند، و آداپتورهای رویداد، رویدادهای منتشر شده توسط business logic را منتشر می‌کنند.

در فصل 12، زمانی که در مورد فناوری‌های استقرار بحث می‌کنم، خواهید دید که implementation view یک service می‌تواند اشکال متعددی داشته باشد. مؤلفه ممکن است یک فرآیند مستقل، یک web application یا یک بسته OSGI باشد که در یک container یا یک تابع ابری serverless اجرا می‌شود. با این حال، یک الزام اساسی این است که یک service دارای یک API است و به طور مستقل قابل استقرار است.

WHAT IS LOOSE COUPLING?

یک ویژگی مهم معماری microservice این است که services loosely coupled هستند (https://en.wikipedia.org/wiki/Loose_coupling). تمام تعامل با یک service از طریق API آن انجام می‌شود، که جزئیات پیاده‌سازی آن را کپسوله می‌کند. این امر implementation service را قادر می‌سازد تا بدون تأثیر بر clients آن، تغییر کند. Services loosely coupled برای بهبود ویژگی‌های زمان توسعه یک application، از جمله قابلیت نگهداری و قابلیت تست آن، کلیدی هستند. درک، تغییر و تست آن‌ها بسیار آسان‌تر است.

الزامی برای اینکه services loosely coupled باشند و فقط از طریق APIs با هم همکاری کنند، services را از برقراری ارتباط از طریق یک پایگاه داده منع می‌کند. شما باید با داده‌های پایدار یک service مانند فیلدهای یک کلاس رفتار کنید و آن‌ها را خصوصی نگه دارید. خصوصی نگه داشتن داده‌ها developer را قادر می‌سازد تا طرحواره پایگاه داده service خود را بدون نیاز به

شکل 2.4

یک service دارای یک API است که implementation را کپسوله می‌کند. API عملیات را تعریف می‌کند که توسط clients فراخوانی می‌شوند. دو نوع عملیات وجود دارد: دستورات داده‌ها را به‌روزرسانی می‌کنند، و queries داده‌ها را بازیابی می‌کنند. هنگامی که داده‌های آن تغییر می‌کند، یک service رویدادهایی را منتشر می‌کند که clients می‌توانند در آن‌ها مشترک شوند.
