WHY ARCHITECTURE MATTERS

یک application دارای دو دسته از الزامات است. دسته اول شامل الزامات عملکردی است که تعریف می‌کنند application باید چه کاری انجام دهد. آن‌ها معمولاً به شکل موارد استفاده یا user stories هستند. معماری ارتباط بسیار کمی با الزامات عملکردی دارد. شما می‌توانید الزامات عملکردی را تقریباً با هر معماری، حتی یک توپ بزرگ گل، پیاده‌سازی کنید.

معماری مهم است زیرا application را قادر می‌سازد تا دسته دوم الزامات را برآورده کند: الزامات کیفیت service آن. اینها به عنوان ویژگی‌های کیفیت و به اصطلاح -ilities نیز شناخته می‌شوند. الزامات کیفیت service، کیفیت‌های زمان اجرا مانند مقیاس‌پذیری و قابلیت اطمینان را تعریف می‌کنند. آن‌ها همچنین کیفیت‌های زمان توسعه از جمله قابلیت نگهداری، قابلیت تست و قابلیت استقرار را تعریف می‌کنند. معماری که برای application خود انتخاب می‌کنید، تعیین می‌کند که چقدر این الزامات کیفیتی را برآورده می‌کند.

Overview of architectural styles

در دنیای فیزیکی، معماری یک ساختمان اغلب از یک سبک خاص، مانند ویکتوریایی، American Craftsman یا Art Deco پیروی می‌کند. هر سبک بسته‌ای از تصمیمات طراحی است که ویژگی‌های ساختمان و مصالح ساختمانی را محدود می‌کند. مفهوم سبک معماری برای software نیز صدق می‌کند. David Garlan و Mary Shaw (An Introduction to Software Architecture, January 1994, https://www.cs.cmu.edu/afs/cs/project/able/ftp/intro_softarch/intro_softarch.pdf)، پیشگامان در رشته معماری software، یک سبک معماری را به شرح زیر تعریف می‌کنند:

بنابراین، یک سبک معماری، یک خانواده از این سیستم‌ها را از نظر الگوی سازماندهی ساختاری تعریف می‌کند. به‌طور خاص‌تر، یک سبک معماری، واژگان مؤلفه‌ها و اتصالات را که می‌توان در نمونه‌هایی از آن سبک استفاده کرد، به همراه مجموعه‌ای از محدودیت‌ها در مورد نحوه ترکیب آن‌ها تعیین می‌کند.

یک سبک معماری خاص، پالت محدودی از عناصر (مؤلفه‌ها) و روابط (اتصالات) را ارائه می‌دهد که می‌توانید view معماری application خود را از آن تعریف کنید. یک application معمولاً از ترکیبی از سبک‌های معماری استفاده می‌کند. به عنوان مثال، در ادامه این بخش نحوه عملکرد معماری monolithic را شرح می‌دهم که سبک معماری است که implementation view را به عنوان یک مؤلفه واحد (قابل اجرا/قابل استقرار) ساختار می‌دهد. معماری microservice یک application را به عنوان مجموعه‌ای از services loosely coupled ساختار می‌دهد.

THE LAYERED ARCHITECTURAL STYLE

نمونه کلاسیک یک سبک معماری، معماری لایه‌ای است. یک معماری لایه‌ای عناصر software را به لایه‌ها سازماندهی می‌کند. هر لایه مجموعه‌ای از مسئولیت‌های تعریف شده دارد. یک معماری لایه‌ای همچنین وابستگی‌ها را بین لایه‌ها محدود می‌کند. یک لایه فقط می‌تواند به لایه بلافاصله زیر آن (اگر لایه‌بندی سخت باشد) یا هر یک از لایه‌های زیر آن وابسته باشد.
