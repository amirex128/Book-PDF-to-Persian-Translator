2.2.3

Defining services by applying the Decompose by subdomain pattern

DDD، همانطور که در کتاب عالی Domain-driven design اثر Eric Evans (Addison-Wesley Professional, 2003) توضیح داده شده است، یک رویکرد برای ساخت applications software پیچیده است که بر توسعه یک مدل domain شی‌گرا متمرکز است. یک مدل domain، دانش در مورد یک domain را در قالبی که می‌تواند برای حل مشکلات در آن domain استفاده شود، ثبت می‌کند. این واژگانی را که توسط تیم استفاده می‌شود، تعریف می‌کند، آنچه DDD آن را زبان همه‌جا می‌نامد. مدل domain به‌طور تنگاتنگی در طراحی و پیاده‌سازی application منعکس می‌شود. DDD دارای دو مفهوم است که هنگام اعمال معماری microservice، فوق‌العاده مفید هستند: زیر domains و bounded contexts.

DDD کاملاً با رویکرد سنتی مدل‌سازی سازمانی که یک مدل واحد برای کل سازمان ایجاد می‌کند، متفاوت است. به عنوان مثال، در چنین مدلی، یک تعریف واحد از هر نهاد تجاری، مانند مشتری، سفارش و غیره وجود خواهد داشت. مشکل این نوع مدل‌سازی این است که وادار کردن بخش‌های مختلف یک سازمان به توافق بر سر یک مدل واحد، یک کار بزرگ است. همچنین، این بدان معناست که از دیدگاه یک بخش معین از سازمان، مدل برای نیازهای آن‌ها بیش از حد پیچیده است. علاوه بر این، مدل domain می‌تواند گیج‌کننده باشد زیرا بخش‌های مختلف سازمان ممکن است از یک اصطلاح برای مفاهیم مختلف یا اصطلاحات مختلف برای یک مفهوم یکسان استفاده کنند. DDD از این مشکلات با تعریف چندین مدل domain، که هر کدام دارای یک دامنه صریح هستند، اجتناب می‌کند.

DDD یک مدل domain جداگانه برای هر زیر domain تعریف می‌کند. یک زیر domain بخشی از domain است، اصطلاح DDD برای فضای مشکل application. زیر domains با استفاده از همان رویکردی که برای شناسایی قابلیت‌های کسب‌وکار استفاده می‌شود، شناسایی می‌شوند: کسب‌وکار را تجزیه و تحلیل کنید و حوزه‌های مختلف تخصص را شناسایی کنید. نتیجه نهایی به احتمال زیاد زیر domainsی خواهد بود که شبیه قابلیت‌های کسب‌وکار هستند. نمونه‌هایی از زیر domains در FTGO شامل ثبت سفارش، مدیریت سفارش، مدیریت آشپزخانه، تحویل و امور مالی است. همانطور که می‌بینید، این زیر domains بسیار شبیه به قابلیت‌های کسب‌وکار هستند که قبلاً توضیح داده شد.
