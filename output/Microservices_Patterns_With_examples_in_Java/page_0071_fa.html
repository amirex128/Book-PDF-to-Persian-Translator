What is the microservice architecture exactly?

یک محدودیت کلیدی که توسط معماری microservice تحمیل می‌شود این است که services loosely coupled هستند. در نتیجه، محدودیت‌هایی در مورد نحوه همکاری services وجود دارد.

برای توضیح این محدودیت‌ها، سعی می‌کنم اصطلاح service را تعریف کنم، توضیح دهم که loosely coupled بودن به چه معناست و به شما بگویم که چرا این موضوع مهم است.

WHAT IS A SERVICE?

یک service یک مؤلفه software مستقل و قابل استقرار است که عملکرد مفیدی را پیاده‌سازی می‌کند. شکل 2.4 نمای خارجی یک service را نشان می‌دهد، که در این مثال، Order Service است. یک service دارای یک API است که دسترسی clients خود را به عملکرد آن ارائه می‌دهد. دو نوع عملیات وجود دارد: دستورات و queries. API شامل دستورات، queries و رویدادها است. یک دستور، مانند createOrder()، اقدامات را انجام می‌دهد و داده‌ها را به‌روزرسانی می‌کند. یک query، مانند findOrderById()، داده‌ها را بازیابی می‌کند. یک service همچنین رویدادهایی مانند OrderCreated را منتشر می‌کند که توسط clients آن مصرف می‌شود.

API یک service، پیاده‌سازی داخلی آن را کپسوله می‌کند. برخلاف یک monolith، یک developer نمی‌تواند کدی بنویسد که API آن را دور بزند. در نتیجه، معماری microservice، ماژولار بودن application را اعمال می‌کند.

هر service در معماری microservice، معماری و به‌طور بالقوه، technology stack خود را دارد. اما یک service معمولی دارای یک معماری شش ضلعی است. API آن توسط آداپتورهایی پیاده‌سازی می‌شود که با business logic service تعامل دارند. عملیات
