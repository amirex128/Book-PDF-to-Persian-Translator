فصل 1

فرار از جهنم Monolithic

در این فصل. بقیه زبان pattern شامل گروه‌هایی از الگوها است که راه‌حل‌هایی برای مسائلی هستند که با استفاده از الگوی معماری Microservice معرفی شده‌اند.

الگوها نیز به سه لایه تقسیم می‌شوند:

  • الگوهای زیرساختی - این‌ها مشکلاتی را حل می‌کنند که بیشتر مسائل زیرساختی در خارج از توسعه هستند.
  • زیرساخت Application - اینها برای مسائل زیرساختی هستند که بر توسعه نیز تأثیر می‌گذارند.
  • الگوهای Application - این‌ها مشکلاتی را حل می‌کنند که developers با آن مواجه هستند.

این الگوها بر اساس نوع مشکلی که حل می‌کنند، با هم گروه‌بندی شده‌اند. بیایید به گروه‌های اصلی الگوها نگاهی بیندازیم.

PATTERNS FOR DECOMPOSING AN APPLICATION INTO SERVICES

تصمیم‌گیری در مورد چگونگی تجزیه یک سیستم به مجموعه‌ای از services، یک هنر است، اما تعدادی استراتژی وجود دارد که می‌تواند کمک کند. دو الگوی تجزیه نشان داده شده در شکل 1.11 استراتژی‌های متفاوتی هستند که می‌توانید برای تعریف معماری application خود استفاده کنید.

فصل 2 این الگوها را با جزئیات شرح می‌دهد.

COMMUNICATION PATTERNS

یک application که با استفاده از معماری microservice ساخته شده است، یک سیستم توزیع شده است. در نتیجه، ارتباط بین فرآیندی (IPC) بخش مهمی از معماری microservice است. شما باید تصمیمات مختلف معماری و طراحی را در مورد نحوه ارتباط services خود با یکدیگر و دنیای بیرون اتخاذ کنید. شکل 1.12 الگوهای ارتباطی را نشان می‌دهد، که به پنج گروه سازماندهی شده‌اند:

  • سبک ارتباط - از چه نوع مکانیزم IPC باید استفاده کنید؟
  • کشف - چگونه یک client یک service آدرس IP یک نمونه service را تعیین می‌کند تا، به عنوان مثال، یک درخواست HTTP را انجام دهد؟
  • قابلیت اطمینان - چگونه می‌توانید اطمینان حاصل کنید که ارتباط بین services قابل اعتماد است، حتی اگر services در دسترس نباشند؟
  • پیام‌رسانی تراکنشی - چگونه باید ارسال پیام‌ها و انتشار رویدادها را با database transactions که داده‌های تجاری را به‌روز می‌کنند، ادغام کنید؟
  • API خارجی - چگونه clients application شما با services ارتباط برقرار می‌کنند؟
