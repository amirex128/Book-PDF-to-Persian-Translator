Part Two Quality Attributes

Problem: چگونه می‌توانیم سیستم را به تعدادی از ساختارهای اجرایی مستقل محاسباتی - گروه‌هایی از نرم‌افزار و سخت‌افزار - متصل به یک رسانه ارتباطی تقسیم کنیم؟ این کار برای ارائه محیط‌های server خاص که برای الزامات عملیاتی و استفاده از منابع بهینه شده‌اند، انجام می‌شود.

Solution: ساختارهای اجرایی بسیاری از سیستم‌ها به عنوان مجموعه‌ای از گروه‌بندی‌های منطقی از components سازماندهی شده‌اند. هر گروه‌بندی یک tier نامیده می‌شود. گروه‌بندی components در tiers ممکن است بر اساس معیارهای مختلفی باشد، مانند نوع component، اشتراک محیط اجرایی یکسان، یا داشتن هدف زمان اجرا یکسان.

استفاده از tiers ممکن است در هر مجموعه (یا الگو) از runtime components اعمال شود، اگرچه در عمل، اغلب در زمینه الگوهای client-server استفاده می‌شود. Tiers، محدودیت‌های topological را القا می‌کنند که محدود می‌کنند کدام components ممکن است با سایر components ارتباط برقرار کنند. به‌طور خاص، connectors ممکن است فقط بین components در یک tier یکسان یا در tiers مجاور وجود داشته باشند. الگوی چند لایه که در بسیاری از برنامه‌های Java EE و Microsoft .NET یافت می‌شود، نمونه‌ای از سازمان در tiers است که از الگوی client-server مشتق شده است.

علاوه بر این، tiers ممکن است انواع ارتباطاتی را که می‌توانند در سراسر tiers مجاور انجام شوند، محدود کنند. به عنوان مثال، برخی از الگوهای tiered به ارتباط فراخوانی-بازگشت در یک جهت و notification مبتنی بر رویداد در جهت دیگر نیاز دارند.

ضعف اصلی معماری چند لایه، هزینه و پیچیدگی آن است. برای سیستم‌های ساده، مزایای معماری چند لایه ممکن است هزینه‌های اولیه و مداوم آن را، از نظر سخت‌افزار، نرم‌افزار، و طراحی و پیچیدگی پیاده‌سازی، توجیه نکند.

Tiers، components نیستند، بلکه گروه‌بندی‌های منطقی از components هستند. همچنین، tiers را با لایه‌ها اشتباه نگیرید! Layering یک الگو از ماژول‌ها (یک واحد پیاده‌سازی) است، در حالی که tiers فقط برای موجودیت‌های زمان اجرا اعمال می‌شود.

جدول 13.11 راه‌حل الگوی چند لایه را خلاصه می‌کند.

Tiers، اطمینان از security و بهینه‌سازی performance و availability را به روش‌های تخصصی آسان‌تر می‌کنند. آنها همچنین modifiability سیستم را افزایش می‌دهند، زیرا زیر گروه‌های محاسباتی مستقل باید بر سر پروتکل‌ها برای تعامل توافق کنند، بنابراین coupling آنها را کاهش می‌دهند.

شکل 13.15 از یک نماد غیررسمی برای توصیف معماری چند لایه یک برنامه Java EE وب‌سایت مصرف‌کننده استفاده می‌کند. این برنامه بخشی از سیستم Adventure Builder است. بسیاری از انواع component-and-connector، مختص پلتفرم پشتیبانی‌کننده هستند، که در این مورد، Java EE است.
