7—Modifiability

  • چه زمانی تغییر ایجاد می شود و چه کسی آن را ایجاد می کند؟ در گذشته، یک تغییر معمولاً در سورس کد ایجاد می شد. یعنی یک developer باید تغییر را ایجاد می کرد که آزمایش می شد و سپس در یک نسخه جدید deployed می شد. با این حال، اکنون، سوال زمان ایجاد تغییر با سوال چه کسی آن را ایجاد می کند، در هم تنیده شده است. یک کاربر نهایی که محافظ صفحه را تغییر می دهد، به وضوح در حال ایجاد تغییر در یکی از جنبه های سیستم است. به همان اندازه واضح است که این در همان دسته تغییر سیستم به گونه ای نیست که بتواند از طریق وب استفاده شود نه روی یک ماشین واحد. تغییرات را می توان در implementation (با اصلاح سورس کد)، در حین compile (با استفاده از سوئیچ های زمان compile)، در طول build (با انتخاب کتابخانه ها)، در طول configuration (توسط طیف وسیعی از تکنیک ها، از جمله تنظیم پارامتر) یا در حین execution (توسط تنظیمات پارامتر، plugins و غیره) ایجاد کرد. یک تغییر همچنین می تواند توسط یک developer، یک کاربر نهایی یا یک system administrator ایجاد شود.
  • هزینه تغییر چقدر است؟ ایجاد یک سیستم بیشتر modifiable شامل دو نوع هزینه است:
    • هزینه معرفی مکانیسم (ها) برای modifiable کردن سیستم
    • هزینه ایجاد تغییر با استفاده از مکانیسم (ها)
    به عنوان مثال، ساده ترین مکانیسم برای ایجاد تغییر این است که منتظر بمانید تا یک درخواست تغییر (change request) وارد شود، سپس سورس کد را برای انطباق با درخواست تغییر دهید. هزینه معرفی مکانیسم صفر است. هزینه اعمال آن، هزینه تغییر سورس کد و اعتبارسنجی مجدد سیستم است. در انتهای دیگر این طیف، یک تولید کننده application، مانند یک user interface سازنده، قرار دارد. سازنده، ورودی توصیفی از designer user interface تولید شده از طریق تکنیک های دستکاری مستقیم را می گیرد و (معمولاً) سورس کد تولید می کند. هزینه معرفی مکانیسم، هزینه ساخت UI builder است که می تواند قابل توجه باشد. هزینه استفاده از مکانیسم، هزینه تولید ورودی برای تغذیه سازنده (هزینه می تواند قابل توجه یا ناچیز باشد)، هزینه اجرای سازنده (تقریباً صفر) و سپس هزینه هر آزمایشی است که بر روی نتیجه انجام می شود (معمولاً بسیار کمتر از حد معمول).

برای N اصلاح مشابه، یک توجیه ساده شده برای یک مکانیسم تغییر این است که

N × Cost of making the change without the mechanism < Cost of installing the mechanism + (N × Cost of making the change using the mechanism).

N تعداد پیش بینی شده از اصلاحاتی است که از مکانیسم modifiability استفاده می کنند، اما N یک پیش بینی است. اگر تغییرات کمتری نسبت به حد انتظار وارد شود، ممکن است یک مکانیسم تغییر گران قیمت تضمین نشود. علاوه بر این، هزینه ایجاد مکانیسم modifiability می تواند در جای دیگری اعمال شود - در افزودن functionality، در بهبود performance، یا حتی در سرمایه گذاری های غیر نرم افزاری مانند خرید سهام فناوری. همچنین، معادله زمان را در نظر نمی گیرد.
