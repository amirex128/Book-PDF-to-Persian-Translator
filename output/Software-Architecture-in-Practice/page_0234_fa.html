Part Two Quality Attributes

به نرم‌افزار user interface جدا از بقیه سیستم. کاربران اغلب می‌خواهند به داده‌ها از دیدگاه‌های مختلف نگاه کنند، مانند نمودار میله‌ای یا نمودار دایره‌ای. این بازنمایی‌ها باید منعکس‌کننده state فعلی داده‌ها باشند.

Problem: چگونه می‌توان عملکرد user interface را از عملکرد برنامه جدا نگه داشت و هنوز هم به ورودی کاربر یا تغییرات در داده‌های برنامه اصلی پاسخگو بود؟ و چگونه می‌توان چندین views از user interface را ایجاد، نگهداری و هماهنگ کرد، زمانی که داده‌های برنامه اصلی تغییر می‌کند؟

Solution: الگوی model-view-controller (MVC) عملکرد برنامه را به سه نوع components تقسیم می‌کند:

  • یک model، که حاوی داده‌های برنامه است
  • یک view، که بخشی از داده‌های اساسی را نمایش می‌دهد و با کاربر تعامل دارد
  • یک controller، که بین model و view میانجی‌گری می‌کند و اعلان‌های تغییرات state را مدیریت می‌کند

MVC برای همه موقعیت‌ها مناسب نیست. طراحی و پیاده‌سازی سه نوع مجزا از components، همراه با اشکال مختلف تعامل آنها، می‌تواند پرهزینه باشد، و این هزینه ممکن است برای user interfaces نسبتاً ساده منطقی نباشد. همچنین، تطابق بین abstractions MVC و commercial user interface toolkits کامل نیست. view و controller، ورودی و خروجی را از هم جدا می‌کنند، اما این توابع اغلب در widgets جداگانه ترکیب می‌شوند. این ممکن است منجر به یک conceptual mismatch بین معماری و user interface toolkit شود.

جدول 13.3، راه‌حل الگوی MVC را خلاصه می‌کند.

Table 13.3  Model-View-Controller Pattern Solution
Overview الگوی MVC عملکرد سیستم را به سه components تقسیم می‌کند: یک model، یک view و یک controller که بین model و view میانجی‌گری می‌کند.
Elements The model، یک نمایش از داده‌ها یا state برنامه است، و شامل (یا یک interface را ارائه می‌دهد به) منطق برنامه.
The view، یک user interface component است که یا یک نمایش از model را برای کاربر تولید می‌کند یا امکان نوعی ورودی کاربر را فراهم می‌کند، یا هر دو.
The controller، تعامل بین model و view را مدیریت می‌کند، و اقدامات کاربر را به تغییراتی در model یا تغییراتی در view ترجمه می‌کند.
Relations رابطه notifies، نمونه‌هایی از model، view و controller را متصل می‌کند، که عناصر را از تغییرات state مربوطه مطلع می‌کند.
Constraints باید حداقل یک نمونه از هر model، view و controller وجود داشته باشد.
The model component نباید مستقیماً با controller تعامل داشته باشد.
Weaknesses پیچیدگی ممکن است برای user interfaces ساده ارزش نداشته باشد.
The model، view، و controller abstractions ممکن است برای برخی از user interface toolkits مناسب نباشند.
