10.2  Tactics for Testability

  • جایگزینی Component، که به سادگی پیاده‌سازی یک component را با یک پیاده‌سازی متفاوت جایگزین می‌کند که (در مورد testability) ویژگی‌هایی دارد که testing را تسهیل می‌کنند. جایگزینی Component اغلب در build scripts یک سیستم انجام می‌شود.
  • ماکروهای Preprocessor که، در صورت فعال شدن، به کد گزارش‌دهی state گسترش می‌یابند یا عبارات probe را فعال می‌کنند که اطلاعات را برمی‌گردانند یا نمایش می‌دهند، یا کنترل را به یک testing console برمی‌گردانند.
  • Aspects (در برنامه‌های aspect-oriented) که نگرانی‌های متقاطع نحوه گزارش state را مدیریت می‌کنند.

محدود کردن پیچیدگی

Complex software test کردن سخت‌تر است. این به این دلیل است که، با تعریف پیچیدگی، فضای operating state آن بسیار بزرگ است و (با فرض برابر بودن سایر شرایط) بازآفرینی یک state دقیق در یک فضای state بزرگتر از انجام این کار در یک فضای state کوچک‌تر دشوارتر است. از آنجایی که testing فقط در مورد شکست دادن نرم‌افزار نیست، بلکه در مورد یافتن خطا است که باعث شکست شد تا بتوان آن را حذف کرد، ما اغلب نگران تکرارپذیر کردن رفتار هستیم. این دسته دارای سه تاکتیک است:

  • محدود کردن پیچیدگی ساختاری. این تاکتیک شامل اجتناب یا حل وابستگی‌های چرخه‌ای بین components، جداسازی و کپسوله کردن وابستگی‌ها به محیط خارجی و کاهش وابستگی‌ها بین components به طور کلی (به عنوان مثال، کاهش تعداد دسترسی‌های خارجی به داده‌های عمومی یک ماژول) است. در سیستم‌های object-oriented، می‌توانید سلسله مراتب وراثت را ساده کنید: تعداد classes را که یک class از آنها مشتق شده است یا تعداد classes مشتق شده از یک class را محدود کنید. عمق درخت وراثت و تعداد فرزندان یک class را محدود کنید. چندشکلی و فراخوانی‌های پویا را محدود کنید. یک metric ساختاری که به صورت تجربی نشان داده شده است که با testability همبستگی دارد، پاسخ یک class نامیده می‌شود. پاسخ class C، تعداد متدهای C به‌علاوه تعداد متدهای classes دیگر است که توسط متدهای C فراخوانی می‌شوند. کم نگه داشتن این metric می‌تواند testability را افزایش دهد.
    داشتن انسجام بالا، loose coupling و separation of concerns - همه تاکتیک‌های modifiability (به فصل 7 مراجعه کنید) - نیز می‌تواند به testability کمک کند. آنها نوعی از محدود کردن پیچیدگی عناصر معماری هستند با دادن یک کار متمرکز به هر عنصر با تعامل محدود با عناصر دیگر. Separation of concerns می‌تواند به دستیابی به controllability و observability (و همچنین کاهش اندازه فضای state کلی برنامه) کمک کند. Controllability برای ایجاد testing قابل ردیابی حیاتی است، همانطور که رابرت بایندر اشاره کرده است: "یک component که می‌تواند مستقل از دیگران عمل کند، به راحتی قابل کنترل است. . . . با high coupling در میان classes، معمولاً کنترل class تحت test دشوارتر است، بنابراین testability کاهش می‌یابد. . . . اگر قابلیت‌های user interface با عملکردهای اساسی در هم تنیده باشند، test کردن هر عملکرد دشوارتر خواهد بود" [Binder 94].
