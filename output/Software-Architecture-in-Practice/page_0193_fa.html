10.4  Summary

اطمینان از اینکه یک سیستم به راحتی قابل test است، هم از نظر هزینه testing و هم از نظر قابلیت اطمینان سیستم، نتیجه‌بخش است. ابزاری که اغلب برای اجرای tests استفاده می‌شود، test harness است. Test harnesses سیستم‌های نرم‌افزاری هستند که منابع test مانند test cases و زیرساخت test را کپسوله می‌کنند تا استفاده مجدد از tests در سراسر تکرارها آسان باشد و اعمال زیرساخت test به افزایش‌های جدید سیستم آسان باشد. ابزار دیگر، ایجاد test cases قبل از توسعه یک component است، به طوری که توسعه‌دهندگان می‌دانند که component آنها باید کدام tests را پاس کند.

کنترل و مشاهده system state یک class اصلی از تاکتیک‌های testability است. فراهم کردن توانایی انجام تزریق خطا، ثبت system state در بخش‌های کلیدی سیستم، جدا کردن سیستم از محیط آن، و انتزاع منابع مختلف، همگی تاکتیک‌های متفاوتی برای پشتیبانی از کنترل و مشاهده یک سیستم و components آن هستند.

Complex systems به دلیل فضای state بزرگی که محاسبات آنها در آن انجام می‌شود، و به دلیل تعداد زیاد اتصالات بین عناصر سیستم، test کردن دشوار است. در نتیجه، ساده نگه داشتن سیستم، یک class دیگر از تاکتیک‌ها است که از testability پشتیبانی می‌کند.

10.5  For Further Reading

یک مقدمه کلی عالی برای software testing [Beizer 90] است. برای دیدگاهی مدرن‌تر در مورد testing، و از منظر توسعه‌دهنده نرم‌افزار به جای tester، فریمن و پرایس، توسعه مبتنی بر test را در قلمرو object-oriented پوشش می‌دهند [Freeman 09].

برتولینو و استرگینی [Bertolino 96] توسعه‌دهندگان مدل testing هستند که در شکل 10.1 نشان داده شده است.

یین و بیمان [Yin 94] در مورد executable assertions نوشته‌اند. هارتمان [Hartman 10] تکنیکی را برای استفاده از executable assertions به عنوان وسیله‌ای برای تشخیص شرایط رقابت توصیف می‌کند.

برونتینک و ون دورسن [Bruntink 06] در مورد تأثیر ساختار بر testing می‌نویسند.
