7.3  A Design Checklist for Modifiability

به طور کلی، هر چه در چرخه عمر بتوانیم مقادیر را دیرتر bind کنیم، بهتر است. با این حال، قرار دادن مکانیسم ها برای تسهیل آن late binding تمایل دارد گران تر باشد - یک tradeoff دیگر. و بنابراین معادله در صفحه 118 وارد عمل می شود. ما می خواهیم تا حد امکان دیرتر bind کنیم، تا زمانی که مکانیسمی که این امکان را فراهم می کند، مقرون به صرفه باشد.

تاکتیک هایی برای bind کردن مقادیر در زمان compile یا زمان build عبارتند از:

  • جایگزینی کامپوننت (به عنوان مثال، در یک build script یا makefile)
  • Compile-time parameterization
  • Aspects

تاکتیک هایی برای bind کردن مقادیر در زمان deployment شامل موارد زیر است:

  • Configuration-time binding

تاکتیک هایی برای bind کردن مقادیر در زمان راه اندازی یا مقداردهی اولیه عبارتند از:

  • Resource files

تاکتیک هایی برای bind کردن مقادیر در زمان اجرا عبارتند از:

  • Runtime registration
  • Dynamic lookup (به عنوان مثال، برای services)
  • Interpret parameters
  • Startup time binding
  • Name servers
  • Plug-ins
  • Publish-subscribe
  • Shared repositories
  • Polymorphism

جدا کردن ساخت یک مکانیسم برای modifiability از استفاده از مکانیسم برای ایجاد یک اصلاح، امکان درگیر شدن ذینفعان مختلف را می پذیرد - یک ذینفع (معمولاً یک developer) برای ارائه مکانیسم و ​​یک ذینفع دیگر (به عنوان مثال، یک installer، یا یک کاربر) تا بعداً، احتمالاً در یک فاز کاملاً متفاوت چرخه عمر، آن را اعمال کند. نصب یک مکانیسم به گونه ای که شخص دیگری بتواند تغییری در سیستم ایجاد کند بدون اینکه نیازی به تغییر هیچ کدی داشته باشد، گاهی اوقات externalizing the change نامیده می شود.
