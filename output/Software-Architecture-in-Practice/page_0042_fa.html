جدا کردن نگرانی‌ها. Modules پنهان‌کننده اطلاعات باید مواردی را که احتمال تغییر آنها وجود دارد، کپسوله کنند، بنابراین نرم‌افزار را از اثرات آن تغییرات عایق می‌کند. هر Module باید یک Interface با تعریف خوب داشته باشد که جنبه‌های قابل تغییر را از سایر نرم‌افزارهایی که از امکانات آن استفاده می‌کنند، کپسوله یا "پنهان" می‌کند. این Interfaces باید به تیم‌های توسعه مربوطه اجازه دهد تا تا حد زیادی مستقل از یکدیگر کار کنند.

2. مگر اینکه الزامات شما بی‌سابقه باشد—امکان‌پذیر است، اما بعید است—Quality Attributes شما باید با استفاده از Architectural Patterns و تاکتیک‌های شناخته شده (شرح داده شده در فصل 13) که مختص هر Attribute هستند، به دست آیند.

3. Architecture هرگز نباید به یک نسخه خاص از یک محصول یا ابزار تجاری وابسته باشد. اگر باید، باید طوری ساختاردهی شود که تغییر به نسخه دیگری ساده و ارزان باشد.

4. Modules که داده تولید می‌کنند باید از Modules که داده مصرف می‌کنند جدا شوند. این امر تمایل به افزایش قابلیت اصلاح دارد زیرا تغییرات اغلب به سمت تولید یا مصرف داده محدود می‌شود. اگر داده‌های جدیدی اضافه شود، هر دو طرف باید تغییر کنند، اما جداسازی امکان ارتقاء مرحله‌ای (افزایشی) را فراهم می‌کند.

5. انتظار نداشته باشید که یک مکاتبه یک به یک بین Modules و Components وجود داشته باشد. به عنوان مثال، در سیستم‌هایی با همزمانی، ممکن است نمونه‌های متعددی از یک Component وجود داشته باشد که به موازات هم اجرا می‌شوند، که در آن هر Component از همان Module ساخته شده است. برای سیستم‌هایی با چندین رشته همزمانی، هر رشته ممکن است از Services از چندین Component استفاده کند که هر کدام از یک Module متفاوت ساخته شده‌اند.

6. هر فرآیند باید طوری نوشته شود که اختصاص آن به یک پردازنده خاص به راحتی قابل تغییر باشد، شاید حتی در زمان اجرا.

7. Architecture باید دارای تعداد کمی روش برای تعامل Components باشد. یعنی سیستم باید کارهای مشابه را به روشی مشابه در سراسر انجام دهد. این امر به قابلیت درک، کاهش زمان توسعه، افزایش قابلیت اطمینان و افزایش قابلیت اصلاح کمک می‌کند.

8. Architecture باید شامل مجموعه‌ای خاص (و کوچک) از مناطق رقابت منابع باشد، که راه‌حل آن به وضوح مشخص و نگهداری می‌شود. به عنوان مثال، اگر استفاده از شبکه یک نگرانی است، Architect باید برای هر تیم توسعه، دستورالعمل‌هایی را تولید (و اعمال) کند که منجر به حداقل ترافیک شبکه شود. اگر عملکرد یک نگرانی است، Architect باید بودجه زمانی را برای رشته‌های اصلی تولید (و اعمال) کند.

1.5  خلاصه

Software Architecture یک سیستم، مجموعه‌ای از ساختارهایی است که برای استدلال در مورد سیستم مورد نیاز است، که شامل عناصر نرم‌افزاری، روابط بین آنها و ویژگی‌های هر دو است.
