10—Testability

Testing کد یک مورد خاص از validation است، که اطمینان حاصل کردن از این است که یک engineered artifact نیازهای ذینفعان خود را برآورده می‌کند یا برای استفاده مناسب است. در فصل 21 ما در مورد بررسی‌های طراحی معماری بحث خواهیم کرد. این نوع دیگری از validation است، که در آن artifact که در حال test است، معماری است. در این فصل ما فقط با testability یک سیستم در حال اجرا و کد منبع آن سروکار داریم.

10.1  Testability General Scenario

اکنون می‌توانیم سناریوی کلی برای testability را توصیف کنیم.

  • منبع stimulus. Testing توسط unit testers، integration testers، یا system testers (در سمت سازمان توسعه‌دهنده) یا acceptance testers و کاربران نهایی (در سمت مشتری) انجام می‌شود. منبع می‌تواند انسان یا یک automated tester باشد.
  • Stimulus. مجموعه‌ای از tests به دلیل تکمیل یک coding increment مانند یک لایه یا service class، تکمیل یکپارچه‌سازی یک زیرسیستم، پیاده‌سازی کامل کل سیستم، یا تحویل سیستم به مشتری اجرا می‌شود.
  • Artifact. یک واحد کد (متناظر با یک ماژول در معماری)، یک زیرسیستم، یا کل سیستم، artifact است که در حال test است.
  • Environment. Test می‌تواند در زمان توسعه، زمان کامپایل، زمان استقرار، یا در حالی که سیستم در حال اجرا است (شاید در استفاده معمول) رخ دهد. Environment همچنین می‌تواند شامل test harness یا محیط‌های test مورد استفاده باشد.
  • Response. سیستم را می‌توان کنترل کرد تا tests مورد نظر را انجام دهد و نتایج حاصل از test را می‌توان مشاهده کرد.
  • Response measure. اقدامات پاسخگویی با هدف نشان دادن این هستند که یک سیستم تحت test چقدر به راحتی خطاهای خود را "تسلیم" می‌کند. اندازه‌گیری‌ها ممکن است شامل تلاش انجام شده در یافتن یک خطا یا یک نوع خاص از خطاها، تلاش مورد نیاز برای test درصد معینی از دستورات، طولانی‌ترین زنجیره test (معیاری از دشواری انجام tests)، اقدامات تلاش برای انجام tests، اقدامات تلاش برای یافتن خطاها، برآورد احتمال یافتن خطاهای اضافی و مدت زمان یا میزان تلاش برای آماده‌سازی محیط test باشد.
    شاید یک معیار سهولت درآوردن سیستم به یک state خاص باشد. علاوه بر این، می‌توان از اقدامات کاهش خطر خطاهای باقی‌مانده در سیستم استفاده کرد. همه خطاها از نظر تأثیر احتمالی‌شان برابر نیستند. اقدامات کاهش خطر، تلاش می‌کنند تا شدت خطاهای یافت شده (یا یافتنی) را ارزیابی کنند.

شکل 10.2 یک سناریوی مشخص برای testability را نشان می‌دهد. unit tester یک واحد کد را در طول توسعه تکمیل می‌کند و یک توالی test را انجام می‌دهد که نتایج آن ثبت می‌شود و پوشش مسیر 85 درصدی را در عرض سه ساعت testing ارائه می‌دهد.
