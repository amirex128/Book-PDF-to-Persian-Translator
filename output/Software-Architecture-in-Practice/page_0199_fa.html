11—Usability

جدا کردن User Interface!

یکی از مفیدترین کارهایی که یک architect می‌تواند برای قابل استفاده کردن یک سیستم انجام دهد، تسهیل آزمایش با user interface از طریق ساخت نمونه‌های اولیه سریع است. ساخت یک نمونه اولیه، یا چندین نمونه اولیه، برای اینکه به کاربران واقعی اجازه دهید رابط را تجربه کنند و بازخورد آنها را ارائه دهید، سود زیادی دارد. بهترین راه برای انجام این کار این است که نرم‌افزار را طوری طراحی کنید که user interface را بتوان به سرعت تغییر داد.

تاکتیک‌های modifiability که در فصل 7 دیدیم، کاملاً از این هدف پشتیبانی می‌کنند، به‌ویژه موارد زیر:

  • افزایش semantic coherence، encapsulate کردن و co-locate کردن مسئولیت‌های مرتبط، که مسئولیت‌های user interface را در یک مکان واحد قرار می‌دهد.
  • محدود کردن وابستگی‌ها، که اثر موجی را بر سایر نرم‌افزارها هنگام تغییر user interface به حداقل می‌رساند.
  • Defer binding، که به شما امکان می‌دهد انتخاب‌های مهم user interface را بدون نیاز به recode انجام دهید.

Defer binding در اینجا بسیار مفید است، زیرا می‌توانید انتظار داشته باشید که user interface محصول شما در طول testing و حتی پس از عرضه به بازار تحت فشار تغییر قرار گیرد.

ابزارهای تولید user interface با این تاکتیک‌ها سازگار هستند. اکثر آنها یک ماژول واحد با یک abstract interface برای بقیه نرم‌افزار تولید می‌کنند. بسیاری از آنها قابلیت تغییر user interface را پس از زمان کامپایل فراهم می‌کنند. در صورت تصمیم به استفاده از ابزار دیگری، می‌توانید با محدود کردن وابستگی‌ها به ماژول تولید شده، سهم خود را انجام دهید.

کارهای زیادی در الگوهای مختلف جداسازی user interface در دهه‌های 1980 و 90 انجام شد. با ظهور وب و مدرن‌سازی الگوی model-view-controller (MVC) برای بازتاب رابط‌های وب، MVC به الگوی جداسازی غالب تبدیل شده است. اکنون الگوی MVC در طیف گسترده‌ای از فریم‌ورک‌های مختلف ساخته شده است. (به فصل 14 برای بحث در مورد MVC مراجعه کنید.) MVC ارائه دیدگاه‌های متعدد از داده‌ها را آسان می‌کند و از ابتکار عمل کاربر پشتیبانی می‌کند، همانطور که در ادامه بحث می‌کنیم.

بسیاری از اوقات quality attributes با یکدیگر در تضاد هستند. از طرف دیگر، Usability و modifiability اغلب یکدیگر را تکمیل می‌کنند، زیرا یکی از بهترین راه‌ها برای افزایش قابلیت استفاده یک سیستم، modifiability آن است. با این حال، این همیشه صادق نیست. در بسیاری از سیستم‌ها، قوانین کسب‌وکار، UI را هدایت می‌کنند - به عنوان مثال، مشخص کردن نحوه تأیید اعتبار ورودی. برای تحقق این اعتبار، UI ممکن است نیاز به فراخوانی یک سرور داشته باشد (که می‌تواند بر performance تأثیر منفی بگذارد). برای دور زدن این performance penalty، architect ممکن است انتخاب کند که این قوانین را در کلاینت و سرور تکرار کند، که در این صورت تکامل را دشوار می‌کند. افسوس، زندگی architect هرگز آسان نیست!
