10—Testability

شکل 10.1  مدلی از testing

برای اینکه یک سیستم به درستی قابل test باشد، باید بتوان ورودی‌های هر component را کنترل کرد (و احتمالاً internal state آن را دستکاری کرد) و سپس خروجی‌های آن (و احتمالاً internal state آن، چه بعد از محاسبه خروجی‌ها و چه در مسیر محاسبه آنها) را مشاهده کرد. این کنترل و مشاهده اغلب از طریق استفاده از یک test harness انجام می‌شود که نرم‌افزار تخصصی (یا در برخی موارد، سخت‌افزار) است که برای آزمایش نرم‌افزار تحت test طراحی شده است. Test harnesses در اشکال مختلفی وجود دارند، مانند قابلیت ضبط و پخش برای داده‌های ارسال شده در سراسر رابط‌های مختلف، یا یک شبیه‌ساز برای یک محیط خارجی که در آن یک قطعه نرم‌افزار تعبیه‌شده آزمایش می‌شود، یا حتی در حین تولید (به بخش فرعی مراجعه کنید). Test harness می‌تواند در اجرای روش‌های test و ثبت خروجی کمک کند. یک test harness می‌تواند یک قطعه نرم‌افزاری اساسی باشد که در حق خود، معماری، ذینفعان و الزامات quality attribute خود را دارد.

Testing توسط توسعه‌دهندگان، کاربران یا پرسنل تضمین کیفیت انجام می‌شود. ممکن است بخش‌هایی از سیستم یا کل سیستم test شوند. اقدامات پاسخگویی برای testability به این می‌پردازد که tests چقدر در کشف خطاها موثر هستند و چقدر طول می‌کشد تا tests را تا سطح پوشش مورد نظر انجام دهیم. Test cases می‌تواند توسط توسعه‌دهندگان، گروه testing یا مشتری نوشته شود. Test cases می‌تواند بخشی از acceptance testing باشد یا می‌تواند توسعه را هدایت کند، همانطور که در انواع خاصی از روش‌های Agile انجام می‌دهند.
