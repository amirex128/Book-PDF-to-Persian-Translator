1—What Is Software Architecture?

چشم‌اندازهای مختلف سیستم، آنها مستقل نیستند. عناصر یک Structure، با عناصر Structures دیگر مرتبط خواهند بود، و ما باید در مورد این روابط استدلال کنیم. به عنوان مثال، یک Module در یک ساختار تجزیه ممکن است به عنوان یک، بخشی از یک، یا چندین Component در یکی از ساختارهای component-and-connector، که Runtime Alter Ego آن را منعکس می‌کند، نشان داده شود. به‌طور کلی، نگاشت‌ها بین Structures چند به چند هستند.

شکل 1.2 نمونه‌ای بسیار ساده از نحوه ارتباط دو Structure با یکدیگر را نشان می‌دهد. شکل سمت چپ، یک Module Decomposition View از یک سیستم Client-Server کوچک را نشان می‌دهد. در این سیستم، دو Module باید Implemented شوند: نرم‌افزار Client و نرم‌افزار Server. شکل سمت راست، یک Component-and-connector View از همان سیستم را نشان می‌دهد. در زمان اجرا ده Client در حال اجرا و دسترسی به Server هستند. بنابراین، این سیستم کوچک دارای دو Module و یازده Component (و ده Connector) است.

در حالی که مکاتبات بین عناصر در ساختار تجزیه و ساختار Client-Server واضح است، این دو View برای موارد بسیار متفاوتی استفاده می‌شوند. به عنوان مثال، View سمت راست می‌تواند برای تجزیه و تحلیل عملکرد، پیش‌بینی Bottleneck و مدیریت ترافیک شبکه استفاده شود، که انجام آن با View سمت چپ بسیار دشوار یا غیرممکن خواهد بود.

(در فصل 13 ما در مورد Map-Reduce Pattern یاد خواهیم گرفت، که در آن کپی‌هایی از عملکرد ساده و یکسان در سراسر صدها یا هزاران Processing Nodes توزیع می‌شوند—یک Module برای کل سیستم، اما یک Component در هر Node.)
