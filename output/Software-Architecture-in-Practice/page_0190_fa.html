10.3  A Design Checklist for Testability

جدول 10.2 یک checklist برای پشتیبانی از فرآیند طراحی و تحلیل برای testability است.

Category Checklist
Allocation of Responsibilities تعیین کنید که کدام system responsibilities حیاتی‌ترین هستند و از این رو باید به‌طور کامل آزمایش شوند. اطمینان حاصل کنید که system responsibilities اضافی برای انجام موارد زیر اختصاص داده شده‌اند:
  • اجرای مجموعه test و ثبت نتایج (external test یا self-test)
  • ثبت (log) فعالیتی که منجر به یک خطا شد یا منجر به رفتار غیرمنتظره (شاید پدیدار شده) شد که لزوماً یک خطا نبود
  • کنترل و مشاهده system state مرتبط برای testing
اطمینان حاصل کنید که تخصیص عملکرد، انسجام بالا، low coupling، strong separation of concerns و پیچیدگی ساختاری کم را فراهم می‌کند.
Coordination Model اطمینان از مکانیسم‌های هماهنگی و ارتباط سیستم:
  • از اجرای یک مجموعه test و ثبت نتایج در داخل یک سیستم یا بین سیستم‌ها پشتیبانی کنید
  • پشتیبانی از ثبت فعالیتی که منجر به یک خطا در داخل یک سیستم یا بین سیستم‌ها شده است
  • پشتیبانی از تزریق و نظارت بر state به کانال‌های ارتباطی برای استفاده در testing، در داخل یک سیستم یا بین سیستم‌ها
  • nondeterminism غیرضروری را معرفی نکنید
Data Model تعیین data abstractions اصلی که باید برای اطمینان از عملکرد صحیح سیستم test شوند.
  • اطمینان حاصل کنید که می‌توان مقادیر نمونه‌هایی از این data abstractions را ثبت کرد
  • اطمینان حاصل کنید که مقادیر نمونه‌هایی از این data abstractions را می‌توان هنگام تزریق state به سیستم تنظیم کرد، به طوری که system state منجر به یک خطا ممکن است بازآفرینی شود
  • اطمینان حاصل کنید که ایجاد، مقداردهی اولیه، تداوم، دستکاری، ترجمه و تخریب نمونه‌هایی از این data abstractions می‌تواند اعمال و ثبت شود
Mapping among Architectural Elements تعیین کنید که چگونه mappings ممکن از عناصر معماری (به ویژه mappings از فرآیندها به پردازنده‌ها، threads به فرآیندها، و ماژول‌ها به components) را test کنید تا پاسخ test مورد نظر به دست آید و شرایط رقابتی بالقوه شناسایی شود. علاوه بر این، تعیین کنید که آیا test کردن mappings غیرقانونی از عناصر معماری امکان‌پذیر است یا خیر.
