Client-Server Pattern

Context: منابع و services مشترکی وجود دارد که تعداد زیادی از clients توزیع‌شده می‌خواهند به آنها دسترسی داشته باشند، و ما می‌خواهیم دسترسی یا کیفیت service را کنترل کنیم.

Problem: با مدیریت مجموعه‌ای از منابع و services مشترک، می‌توانیم modifiability و reuse را با فاکتور کردن services مشترک و مجبور بودن به تغییر آنها در یک مکان واحد، یا تعداد کمی از مکان‌ها، ارتقا دهیم. ما می‌خواهیم scalability و availability را با متمرکز کردن کنترل این منابع و services، در حالی که خود منابع را در چندین سرور فیزیکی توزیع می‌کنیم، بهبود بخشیم.

Solution: Clients با درخواست services از servers، که مجموعه‌ای از services را ارائه می‌دهند، تعامل دارند. برخی از components ممکن است به عنوان clients و servers عمل کنند. ممکن است یک server مرکزی یا چندین سرور توزیع‌شده وجود داشته باشد.

راه‌حل الگوی client-server در جدول 13.5 خلاصه شده است. انواع component clients و servers هستند. نوع connector اصلی برای الگوی client-server، یک data connector است که توسط یک پروتکل request/reply برای فراخوانی services هدایت می‌شود.

برخی از معایب الگوی client-server این است که server می‌تواند یک گلوگاه performance باشد و می‌تواند یک نقطه شکست واحد باشد. همچنین، تصمیم‌گیری در مورد مکان قرار دادن عملکرد (در client یا در server) اغلب پیچیده است و تغییر آن پس از ساخت یک سیستم، هزینه دارد.

Table 13.5  Client-Server Pattern Solution
Overview الگوی client-server عملکرد سیستم را به سه components تقسیم می‌کند: یک model، یک view و یک controller که بین model و view میانجی‌گری می‌کند.
Elements The model، یک نمایش از داده‌ها یا state برنامه است، و شامل (یا یک interface را ارائه می‌دهد به) منطق برنامه.
The view، یک user interface component است که یا یک نمایش از model را برای کاربر تولید می‌کند یا امکان نوعی ورودی کاربر را فراهم می‌کند، یا هر دو.
The controller، تعامل بین model و view را مدیریت می‌کند، و اقدامات کاربر را به تغییراتی در model یا تغییراتی در view ترجمه می‌کند.
Relations رابطه notifies، نمونه‌هایی از model، view و controller را متصل می‌کند، که عناصر را از تغییرات state مربوطه مطلع می‌کند.
Constraints باید حداقل یک نمونه از هر model، view و controller وجود داشته باشد.
The model component نباید مستقیماً با controller تعامل داشته باشد.
Weaknesses پیچیدگی ممکن است برای user interfaces ساده ارزش نداشته باشد.
The model، view، و controller abstractions ممکن است برای برخی از user interface toolkits مناسب نباشند.
