10—Testability

10.2  Tactics for Testability

هدف از تاکتیک‌های testability این است که testing را آسان‌تر کند. شکل 10.3 استفاده از تاکتیک‌های testability را نشان می‌دهد. تکنیک‌های معماری برای افزایش software testability به اندازه رشته‌های quality attribute بالغ‌تر مانند modifiability، performance و availability، مورد توجه قرار نگرفته‌اند، اما همانطور که قبلاً گفتیم، هر کاری که architect بتواند برای کاهش هزینه بالای testing انجام دهد، سود قابل توجهی خواهد داشت.

دو دسته از تاکتیک‌ها برای testability وجود دارد. دسته اول به افزودن controllability و observability به سیستم می‌پردازد. دسته دوم به محدود کردن پیچیدگی در طراحی سیستم می‌پردازد.

کنترل و مشاهده System State

کنترل و مشاهده آنقدر برای testability مهم است که برخی از نویسندگان حتی testability را با این اصطلاحات تعریف می‌کنند. این دو با هم پیش می‌روند. کنترل چیزی اگر نتوانید آنچه را که هنگام انجام این کار رخ می‌دهد مشاهده کنید، منطقی نیست. ساده‌ترین شکل کنترل و مشاهده، ارائه یک software component با مجموعه‌ای از ورودی‌ها، اجازه دادن به آن برای انجام کار خود و سپس مشاهده خروجی‌های آن است. با این حال، دسته کنترل و مشاهده system state از تاکتیک‌های testability، بینشی را در مورد نرم‌افزاری ارائه می‌دهد که فراتر از ورودی‌ها و خروجی‌های آن است. این تاکتیک‌ها باعث می‌شوند یک component نوعی از اطلاعات state را حفظ کند، به testers اجازه می‌دهد مقداری را به آن اطلاعات state اختصاص دهند، و/یا آن اطلاعات را در صورت تقاضا در دسترس testers قرار دهند. اطلاعات state ممکن است یک operating state، مقدار برخی از متغیرهای کلیدی، performance load، مراحل میانی فرآیند، یا هر چیز دیگری که برای بازآفرینی رفتار component مفید است، باشد. تاکتیک‌های خاص شامل موارد زیر است:
