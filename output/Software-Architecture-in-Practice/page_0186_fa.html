10.2  Tactics for Testability

  • Specialized interfaces. داشتن testing interfaces تخصصی به شما این امکان را می‌دهد که مقادیر متغیرها را برای یک component یا از طریق یک test harness یا از طریق اجراهای عادی کنترل یا ضبط کنید. نمونه‌هایی از روال‌های test تخصصی شامل موارد زیر است:
    • یک متد set و get برای متغیرها، modes یا attributes مهم (متدهایی که در غیر این صورت ممکن است به جز برای اهداف testing در دسترس نباشند)
    • یک متد report که full state از object را برمی‌گرداند
    • یک متد reset برای تنظیم internal state (به عنوان مثال، تمام attributes یک class) به یک internal state مشخص شده
    • یک متد برای روشن کردن verbose output، سطوح مختلف ثبت رویداد، performance instrumentation، یا resource monitoring
    Specialized testing interfaces و متدها باید به وضوح شناسایی شوند یا از متدهای دسترسی و interfaces برای عملکرد مورد نیاز جدا نگه داشته شوند، تا در صورت نیاز بتوان آنها را حذف کرد. (با این حال، در سیستم‌های performance-critical و برخی از سیستم‌های safety-critical، ارائه کدی متفاوت از آنچه آزمایش شده، مشکل‌ساز است. اگر کد test را حذف کنید، چگونه متوجه می‌شوید که کدی که ارائه می‌دهید رفتار یکسانی دارد، به‌ویژه رفتار زمان‌بندی یکسان، با کدی که test کرده‌اید؟ با این حال، برای انواع دیگر سیستم‌ها، این استراتژی مؤثر است.)
  • Record/playback. State که باعث ایجاد خطا شده است، اغلب بازآفرینی آن دشوار است. ثبت state هنگام عبور از یک interface به آن state اجازه می‌دهد تا از "play the system back" استفاده شود و خطا را دوباره ایجاد کند. Record/playback به هر دو ضبط اطلاعات عبوری از یک interface و استفاده از آن به عنوان ورودی برای testing بیشتر اشاره دارد.
  • Localize state storage. برای راه‌اندازی یک سیستم، زیرسیستم، یا ماژول در یک state دلخواه برای یک test، اگر آن state در یک مکان واحد ذخیره شود، راحت‌ترین حالت است. در مقابل، اگر state دفن یا توزیع شده باشد، این کار دشوار می‌شود، اگر غیرممکن نباشد. State می‌تواند ریزدانه، حتی در سطح بیت، یا درشت‌دانه باشد تا انتزاع‌های گسترده یا operational modes کلی را نشان دهد. انتخاب granularity به نحوه استفاده از states در testing بستگی دارد. یک راه مناسب برای "خارج کردن" state storage (یعنی، قادر ساختن آن به دستکاری از طریق ویژگی‌های interface) استفاده از یک state machine (یا state machine object) به عنوان مکانیسم برای ردیابی و گزارش current state است.
  • Abstract data sources. مشابه کنترل state یک برنامه، کنترل آسان داده‌های ورودی آن، testing را آسان‌تر می‌کند. انتزاع interfaces به شما این امکان را می‌دهد که داده‌های test را راحت‌تر جایگزین کنید. به عنوان مثال، اگر یک پایگاه داده از تراکنش‌های مشتری دارید، می‌توانید معماری خود را طوری طراحی کنید که سیستم test شما به راحتی به پایگاه‌های داده test دیگر، یا احتمالاً حتی به فایل‌های داده‌های test اشاره کند، بدون اینکه نیاز به تغییر کد عملکردی خود داشته باشید.
  • Sandbox. "Sandboxing" به ایزوله کردن یک نمونه از سیستم از دنیای واقعی اشاره دارد تا آزمایش را بدون محدودیت نگرانی
