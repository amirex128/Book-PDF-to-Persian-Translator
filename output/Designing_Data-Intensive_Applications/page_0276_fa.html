یک application یک query ایجاد می‌کند، نتیجه را می‌خواند، شاید query دیگری را بسته به نتیجه query اول ایجاد می‌کند، و به همین ترتیب. Queries و نتایج بین کد application (که روی یک machine اجرا می‌شود) و database server (روی machine دیگری) ارسال می‌شوند.

در این سبک interactive از Transaction، زمان زیادی صرف ارتباطات شبکه بین application و database می‌شود. اگر قرار بود concurrency را در database ممنوع کنید و فقط یک Transaction را در یک زمان پردازش کنید، throughput وحشتناک خواهد بود زیرا database بیشتر وقت خود را صرف انتظار برای صدور query بعدی توسط application برای Transaction فعلی می‌کند. در این نوع database، پردازش چندین Transactions به طور همزمان برای به دست آوردن reasonable performance ضروری است.

به همین دلیل، systems با single-threaded serial transaction processing، interactive multi-statement transactions را مجاز نمی‌دانند. در عوض، application باید کل کد Transaction را از قبل، به عنوان یک stored procedure، به database ارسال کند. تفاوت‌های بین این رویکردها در شکل 7-9 نشان داده شده است. با این شرط که تمام داده‌های مورد نیاز یک Transaction در حافظه باشد، stored procedure می‌تواند بسیار سریع اجرا شود، بدون اینکه منتظر هیچ network یا disk I/O باشد.
