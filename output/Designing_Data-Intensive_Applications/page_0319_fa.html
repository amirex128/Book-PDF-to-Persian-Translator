کاهش داده می‌شود—با تغییر الگوهای تخصیص یا تنظیم تنظیمات GC [66]، ما باید بدترین حالت را در نظر بگیریم اگر می‌خواهیم guarantees های robust ارائه دهیم.

  • در محیط‌های virtualized، یک virtual machine می‌تواند suspended شود (توقف اجرای تمام processes و ذخیره محتویات حافظه در دیسک) و resumed (بازیابی محتویات حافظه و ادامه اجرا). این مکث می‌تواند در هر زمانی در اجرای یک process رخ دهد و می‌تواند برای مدت زمان دلخواه ادامه داشته باشد. این ویژگی گاهی اوقات برای live migration از virtual machines از یک host به دیگری بدون reboot استفاده می‌شود، که در این صورت مدت مکث به میزان نوشتن processes به حافظه بستگی دارد [67].
  • روی دستگاه‌های end-user مانند لپ‌تاپ‌ها، اجرا ممکن است به طور دلخواه suspended و resumed شود، به عنوان مثال، زمانی که user درب لپ‌تاپ خود را می‌بندد.
  • وقتی operating system context-switches به thread دیگری، یا زمانی که hypervisor به یک virtual machine متفاوت سوئیچ می‌کند (هنگام اجرا در یک virtual machine)، thread در حال اجرا می‌تواند در هر نقطه‌ای دلخواه در کد متوقف شود. در مورد یک virtual machine، زمان CPU که در virtual machines های دیگر صرف می‌شود، steal time نامیده می‌شود. اگر machine تحت heavy load است—یعنی اگر صف طولانی از threads در انتظار اجرا وجود داشته باشد—ممکن است مدتی طول بکشد تا thread متوقف شده دوباره اجرا شود.
  • اگر application به synchronous disk access می‌پردازد، یک thread ممکن است منتظر تکمیل یک عملیات slow disk I/O، متوقف شود [68]. در بسیاری از زبان‌ها، disk access می‌تواند به طور شگفت‌انگیزی اتفاق بیفتد، حتی اگر کد به صراحت به دسترسی به فایل اشاره نکند—به عنوان مثال، the Java classloader فایل‌های class را lazily loads می‌کند زمانی که برای اولین بار از آنها استفاده می‌شود، که می‌تواند در هر زمانی در اجرای برنامه اتفاق بیفتد. توقف‌های I/O و GC ممکن است حتی برای ترکیب تأخیرهای خود هم‌دست شوند [69]. اگر دیسک در واقع یک network filesystem یا network block device (مانند EBS آمازون) باشد، I/O latency بیشتر در معرض تغییرپذیری تأخیرهای شبکه قرار دارد [29].
  • اگر operating system برای اجازه دادن به swapping to disk (paging) پیکربندی شده باشد، یک memory access ساده ممکن است منجر به page fault شود که نیاز به بارگذاری یک صفحه از دیسک به حافظه دارد. Thread در حالی که این slow I/O operation در حال انجام است، متوقف می‌شود. اگر memory pressure بالا باشد، این ممکن است به نوبه خود به یک صفحه متفاوت نیاز داشته باشد که به دیسک swapped out شود. در شرایط شدید، operating system ممکن است بیشتر وقت خود را صرف swapping pages in and out of memory کند و کار واقعی کمی انجام دهد (این به عنوان thrashing شناخته می‌شود). برای جلوگیری از این مشکل، paging اغلب روی machine های server غیرفعال می‌شود (اگر شما ترجیح می‌دهید یک process را برای آزاد کردن mem‐ory kill کنید تا اینکه خطر thrashing را بپذیرید).
  • یک Unix process را می‌توان با ارسال سیگنال SIGSTOP متوقف کرد، به عنوان مثال با فشار دادن Ctrl-Z در یک shell. این سیگنال فوراً از دریافت چرخه‌های CPU بیشتر توسط process جلوگیری می‌کند تا زمانی که با SIGCONT از سر گرفته شود، که در آن نقطه اجرای خود را از سر می‌گیرد. حتی اگر محیط شما معمولاً از SIGSTOP استفاده نمی‌کند، ممکن است به طور تصادفی توسط یک مهندس operations ارسال شود.
