iii. با مزاح، مجموعه‌ی queries جستجوی متمایز با نتایج جستجوی غیرتهی، محدود است، با فرض یک corpus محدود. با این حال، در تعداد terms در corpus، که همچنان خبر بدی است، نمایی خواهد بود.

درخواست شما از مجموعه‌داده‌ی مشتق شده می‌خوانید، شاید پردازش بیشتری را روی نتایج انجام می‌دهید، و پاسخ به کاربر را می‌سازید.

در کنار هم، write path و read path کل سفر داده‌ها را در بر می‌گیرند، از نقطه‌ای که جمع‌آوری می‌شود تا نقطه‌ای که مصرف می‌شود (احتمالاً توسط انسان دیگر). Write path بخشی از سفر است که از قبل محاسبه شده است—یعنی، به محض ورود داده‌ها انجام می‌شود، صرف نظر از این‌که کسی از شما خواسته باشد آن را ببیند یا خیر. Read path بخشی از سفر است که فقط زمانی اتفاق می‌افتد که کسی آن را درخواست کند. اگر با زبان‌های برنامه‌نویسی functional آشنا هستید، ممکن است متوجه شوید که write path شبیه به ارزیابی eager، و read path شبیه به ارزیابی lazy است.

مجموعه‌داده‌ی مشتق شده، مکانی است که write path و read path به هم می‌رسند، همان‌طور که در شکل 12-1 نشان داده شده است. این نشان‌دهنده‌ی یک trade-off بین مقدار کاری است که باید در زمان write انجام شود و مقداری که باید در زمان read انجام شود.

Materialized views و caching

یک index جستجوی متن کامل، یک مثال خوب است: write path index را به‌روزرسانی می‌کند، و read path، index را برای keywords جستجو می‌کند. هم reads و هم writes نیاز دارند که کارهایی را انجام دهند. Writes نیاز دارند که ورودی‌های index را برای همه‌ی terms که در سند ظاهر می‌شوند، به‌روزرسانی کنند. Reads نیاز دارند که هر یک از کلمات را در query جستجو کنند، و منطق بولی را برای یافتن اسنادی که شامل همه‌ی کلمات در query (یک AND operator) هستند، یا هر مترادفی از هر یک از کلمات (یک OR operator) اعمال کنند.

اگر شما index نداشتید، یک search query باید تمام اسناد را (مانند grep) اسکن می‌کرد، که اگر تعداد زیادی سند داشتید، بسیار پرهزینه می‌شد. بدون index، کار کمتری در write path (بدون index برای به‌روزرسانی) وجود دارد، اما کار بسیار بیشتری در read path وجود دارد.

از سوی دیگر، شما می‌توانید نتیجه‌ی جستجو را برای همه‌ی queries ممکن از قبل محاسبه کنید. در این حالت، شما کار کمتری را در read path خواهید داشت: بدون منطق بولی، فقط نتایج را برای query خود پیدا کنید و آن‌ها را برگردانید. با این حال، write path بسیار گران‌تر خواهد بود: مجموعه queries جستجوی ممکن که می‌توان درخواست کرد، بی‌نهایت است، و بنابراین محاسبه‌ی قبلی تمام نتایج جستجوی ممکن به زمان و فضای ذخیره‌سازی بی‌نهایت نیاز دارد. این خیلی خوب کار نخواهد کرد.iii

یک گزینه دیگر این است که نتایج جستجو را فقط برای یک مجموعه‌ی ثابت از رایج‌ترین queries از قبل محاسبه کنید، به‌طوری که بتوان آن‌ها را سریعاً بدون نیاز به مراجعه به index سرویس داد.
