که یک Transaction ممکن است بر اساس یک premise منسوخ شده عمل کرده باشد و در آن صورت transaction را abort کند.

چگونه database می‌داند که آیا نتیجه یک query ممکن است تغییر کرده باشد؟ دو مورد وجود دارد که باید در نظر گرفت:

  • تشخیص reads از یک stale MVCC object version (uncommitted write قبل از read رخ داده است)
  • تشخیص writes که بر reads قبلی تأثیر می‌گذارند (write پس از read رخ می‌دهد)

Detecting stale MVCC reads

به یاد بیاورید که snapshot isolation معمولاً با multi-version concurrency control (MVCC) پیاده‌سازی می‌شود (به شکل 7-10 مراجعه کنید). هنگامی که یک Transaction از یک consistent snapshot در یک database MVCC می‌خواند، writes هایی را که توسط هر Transactions دیگری که هنوز در زمان ایجاد snapshot، committed نشده بودند، نادیده می‌گیرد. در شکل 7-10، transaction 43 آلیس را با on_call = true می‌بیند، زیرا transaction 42 (که وضعیت on-call آلیس را تغییر داد) uncommitted است. با این حال، تا زمانی که transaction 43 می‌خواهد commit شود، transaction 42 قبلاً committed شده است. این بدان معناست که write ای که هنگام خواندن از consistent snapshot نادیده گرفته شد، اکنون تأثیر گذاشته است، و premise transaction 43 دیگر درست نیست.
