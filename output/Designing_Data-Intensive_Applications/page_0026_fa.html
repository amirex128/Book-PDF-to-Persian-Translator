اما واقعیت به آن سادگی نیست. بسیاری از سیستم‌های database با ویژگی‌های مختلف وجود دارند، زیرا applications مختلف نیازهای متفاوتی دارند. رویکردهای مختلفی برای caching، چندین روش برای ساخت search indexes و غیره وجود دارد. هنگام ساختن یک application، ما همچنان باید بفهمیم که کدام ابزارها و کدام رویکردها برای کار مورد نظر مناسب‌تر هستند. و ترکیب ابزارها زمانی که نیاز دارید کاری را انجام دهید که یک ابزار واحد به تنهایی نمی‌تواند انجام دهد، می‌تواند دشوار باشد.

این کتاب سفری است از طریق اصول و عملکردهای سیستم‌های data و نحوه استفاده از آن‌ها برای ساخت data-intensive applications. ما بررسی خواهیم کرد که ابزارهای مختلف چه چیز مشترکی دارند، چه چیزی آن‌ها را متمایز می‌کند و چگونه به ویژگی‌های خود می‌رسند.

در این فصل، با بررسی مبانی آنچه که در تلاش برای دستیابی به آن هستیم شروع می‌کنیم: سیستم‌های data قابل اعتماد، مقیاس‌پذیر و قابل نگهداری. ما روشن خواهیم کرد که این موارد به چه معنا هستند، برخی از روش‌های تفکر در مورد آن‌ها را شرح می‌دهیم و به اصول اولیه‌ای که برای فصل‌های بعدی به آن‌ها نیاز خواهیم داشت، می‌پردازیم. در فصل‌های بعدی لایه به لایه ادامه خواهیم داد، و به تصمیمات طراحی مختلفی که باید هنگام کار بر روی یک data-intensive application در نظر گرفته شوند، نگاه خواهیم کرد.

Thinking About Data Systems

ما معمولاً در مورد databases، صف‌ها، caches و غیره به‌عنوان دسته‌های بسیار متفاوتی از ابزارها فکر می‌کنیم. اگرچه یک database و یک صف پیام شباهت‌های سطحی دارند—هر دو data را برای مدتی ذخیره می‌کنند—الگوهای دسترسی آن‌ها بسیار متفاوت است، که به معنای ویژگی‌های عملکردی متفاوت و در نتیجه پیاده‌سازی‌های بسیار متفاوت است.

بنابراین چرا باید همه آن‌ها را تحت یک اصطلاح چتری مانند سیستم‌های data جمع کنیم؟

بسیاری از ابزارهای جدید برای ذخیره‌سازی و پردازش data در سال‌های اخیر پدیدار شده‌اند. آن‌ها برای انواع مختلف موارد استفاده بهینه شده‌اند و دیگر به‌خوبی در دسته‌های سنتی قرار نمی‌گیرند [1]. به‌عنوان‌مثال، datastores وجود دارند که به‌عنوان صف‌های پیام (Redis) نیز استفاده می‌شوند، و صف‌های پیام با تضمین‌های دوام database-like (Apache Kafka) وجود دارند. مرزهای بین دسته‌ها در حال محو شدن هستند.

ثانیاً، به‌طور فزاینده‌ای applications زیادی اکنون نیازهای آن‌قدر زیاد یا گسترده‌ای دارند که یک ابزار واحد دیگر نمی‌تواند تمام نیازهای پردازش و ذخیره‌سازی data خود را برآورده کند. در عوض، کار به وظایفی تقسیم می‌شود که می‌توانند به‌طور مؤثر بر روی یک ابزار واحد انجام شوند و آن ابزارهای مختلف با استفاده از کد application به هم متصل می‌شوند.

به‌عنوان‌مثال، اگر یک لایه caching مدیریت شده توسط application (با استفاده از Memcached یا مشابه) یا یک full-text search server (مانند Elasticsearch یا Solr) جدا از database اصلی خود دارید، معمولاً این مسئولیت کد application است که آن caches و indexes را با database اصلی همگام نگه دارد. شکل 1-1 نگاهی اجمالی به این موضوع می‌دهد (ما در فصل‌های بعدی به جزئیات خواهیم پرداخت).
