iii. این ایده‌آل نیست. اگر اتصال TCP قطع شود، transaction باید aborted شود. اگر interruption پس از درخواست commit توسط client رخ دهد، اما قبل از اینکه server تأیید کند که commit رخ داده است، client نمی‌داند که آیا transaction committed شده است یا خیر. برای حل این مشکل، یک transaction manager می‌تواند operations را با یک unique transaction identifier گروه‌بندی کند که به یک اتصال TCP خاص مرتبط نیست. ما در "The End-to-End Argument for Databases" در صفحه 516 به این موضوع باز خواهیم گشت.

شکل 7-3 نیاز به atomicity را نشان می‌دهد: اگر یک error در جایی در طول transaction رخ دهد، محتویات صندوق پستی و counter خوانده نشده ممکن است از حالت sync خارج شوند. در یک atomic transaction، اگر update به counter شکست بخورد، transaction aborted می‌شود و ایمیل درج شده rolled back می‌شود.

Multi-object transactions نیاز به راهی برای تعیین دارند که کدام operations خواندن و نوشتن به همان transaction تعلق دارند. در relational databases، این کار معمولاً بر اساس اتصال TCP client به database server انجام می‌شود: در هر اتصال خاص، همه چیز بین یک BEGIN TRANSACTION و یک دستور COMMIT به عنوان بخشی از همان transaction در نظر گرفته می‌شود.iii
