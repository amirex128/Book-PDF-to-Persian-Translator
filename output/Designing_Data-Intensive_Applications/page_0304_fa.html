تصور کنید یک system فرضی با یک شبکه که حداکثر تأخیر را برای packets تضمین می‌کند—هر packet یا در زمان d تحویل داده می‌شود، یا از دست می‌رود، اما تحویل هرگز بیشتر از d طول نمی‌کشد. علاوه بر این، فرض کنید که شما می‌توانید تضمین کنید که یک node غیر failed همیشه یک درخواست را در زمان r مدیریت می‌کند. در این حالت، شما می‌توانید تضمین کنید که هر درخواست موفقیت‌آمیز در زمان 2d + r پاسخی دریافت می‌کند—و اگر شما در این مدت پاسخی دریافت نکردید، می‌دانید که یا شبکه یا node از راه دور کار نمی‌کند. اگر این درست بود، 2d + r یک timeout مناسب برای استفاده خواهد بود.

متاسفانه، اکثر systems هایی که ما با آنها کار می‌کنیم، هیچ یک از آن تضمین‌ها را ندارند: شبکه‌های asynchronous، تأخیرهای نامحدود دارند (یعنی، آنها سعی می‌کنند packets را در اسرع وقت تحویل دهند، اما هیچ محدودیتی برای مدت زمانی که ممکن است طول بکشد تا یک packet برسد، وجود ندارد)، و اکثر پیاده‌سازی‌های server نمی‌توانند تضمین کنند که می‌توانند درخواست‌ها را در یک زمان حداکثر مدیریت کنند (به "Response time guarantees" در صفحه 298 مراجعه کنید).

برای تشخیص failure، کافی نیست که system در اکثر مواقع سریع باشد: اگر timeout شما کم باشد، فقط یک transient spike در round-trip times لازم است تا system از تعادل خارج شود.

Network congestion and queueing

هنگام رانندگی با ماشین، زمان سفر در شبکه‌های جاده‌ای اغلب بیشتر به دلیل traffic congestion متفاوت است. به طور مشابه، تغییرپذیری تأخیر packet در شبکه‌های کامپیوتری اغلب به دلیل queueing است [25]:

  • اگر چندین node مختلف به طور همزمان سعی کنند packets را به یک مقصد یکسان ارسال کنند، network switch باید آنها را در صف قرار دهد و آنها را یکی یکی به destination network link تغذیه کند (همانطور که در شکل 8-2 نشان داده شده است). در یک busy network link، یک packet ممکن است مجبور شود مدتی صبر کند تا بتواند یک slot دریافت کند (این network congestion نامیده می‌شود). اگر داده‌های ورودی آنقدر زیاد باشد که صف switch پر شود، packet dropped می‌شود، بنابراین باید دوباره ارسال شود—حتی اگر شبکه به خوبی کار کند.
  • وقتی یک packet به machine مقصد می‌رسد، اگر تمام CPU cores در حال حاضر مشغول باشند، درخواست ورودی از شبکه توسط operating system در صف قرار می‌گیرد تا زمانی که application آماده handling آن شود. بسته به load روی machine، این ممکن است زمان دلخواه را بگیرد.
  • در محیط‌های virtualized، یک operating system در حال اجرا اغلب برای ده‌ها میلی‌ثانیه متوقف می‌شود در حالی که یک virtual machine دیگر از یک CPU core استفاده می‌کند. در این مدت، VM نمی‌تواند هیچ داده‌ای را از شبکه مصرف کند، بنابراین داده‌های ورودی توسط virtual machine monitor در صف قرار می‌گیرند (buffered)، که بیشتر باعث افزایش تغییرپذیری تأخیر شبکه می‌شود.
