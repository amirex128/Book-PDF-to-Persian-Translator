If you were using XSL instead of CSS, you could do something similar: 

<xsl:template match="li[@class='selected']/p">
    <fo:block background-color="blue">
        <xsl:apply-templates/>
    </fo:block>
</xsl:template>

Here, the XPath expression li[@class='selected']/p is equivalent to the CSS selec‐tor li.selected > p in the previous example. What CSS and XSL have in common is that they are both declarative languages for specifying the styling of a document.

تصور کنید اگر مجبور بودید از یک رویکرد imperative استفاده کنید، زندگی چگونه بود. در JavaScript، با استفاده از core Document Object Model (DOM) API، نتیجه ممکن است چیزی شبیه این باشد: 

var liElements = document.getElementsByTagName("li");
for (var i = 0; i < liElements.length; i++) {
    if (liElements[i].className === "selected") {
        var children = liElements[i].childNodes;
        for (var j = 0; j < children.length; j++) {
            var child = children[j];
            if (child.nodeType === Node.ELEMENT_NODE && child.tagName === "P") {
                child.setAttribute("style", "background-color: blue");
            }
        }
    }
}

این JavaScript به طور imperative عنصر <p>Sharks</p> را تنظیم می‌کند تا دارای پس‌زمینه آبی باشد، اما کد وحشتناک است. نه تنها بسیار طولانی‌تر و دشوارتر از معادل‌های CSS و XSL است، بلکه مشکلات جدی نیز دارد:

  • اگر selected class حذف شود (به عنوان مثال، به این دلیل که user روی صفحه دیگری کلیک می‌کند)، رنگ آبی حذف نمی‌شود، حتی اگر کد دوباره اجرا شود — و بنابراین item تا زمانی که کل صفحه دوباره بارگیری شود، برجسته باقی می‌ماند. با CSS، browser به طور خودکار تشخیص می‌دهد که چه زمانی rule li.selected > p دیگر اعمال نمی‌شود و به محض حذف selected class، background آبی را حذف می‌کند.
  • اگر می‌خواهید از یک API جدید، مانند document.getElementsBy ClassName("selected") یا حتی document.evaluate()—که ممکن است performance را بهبود بخشد—استفاده کنید، باید کد را دوباره بنویسید. از سوی دیگر، vendors browser می‌توانند performance of CSS و XPath را بدون شکستن compatibility بهبود بخشند.
