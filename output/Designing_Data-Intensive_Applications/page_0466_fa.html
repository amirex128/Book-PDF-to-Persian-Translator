  • پایگاه‌های داده معمولاً داده‌ها را تا زمانی که صریحاً حذف شوند، نگه می‌دارند، در حالی که اکثر message brokers به‌طور خودکار یک پیام را پس از تحویل موفقیت‌آمیز به consumers آن حذف می‌کنند. چنین message brokers برای ذخیره‌سازی داده‌های بلندمدت مناسب نیستند.
  • از آن‌جایی که آن‌ها پیام‌ها را به‌سرعت حذف می‌کنند، اکثر message brokers فرض می‌کنند که مجموعه کاری آن‌ها نسبتاً کوچک است—یعنی، صف‌ها کوتاه هستند. اگر broker نیاز به buffer کردن تعداد زیادی پیام داشته باشد زیرا consumers کند هستند (شاید اگر دیگر در حافظه جا نشدند، پیام‌ها را روی دیسک spill کنند)، پردازش هر پیام مجزا بیشتر طول می‌کشد، و توان عملیاتی کلی ممکن است کاهش یابد [6].
  • پایگاه‌های داده اغلب از indexes ثانویه و راه‌های مختلفی برای جستجوی داده‌ها پشتیبانی می‌کنند، در حالی که message brokers اغلب از نوعی subscribing به زیرمجموعه‌ای از topics که با برخی الگوها مطابقت دارند، پشتیبانی می‌کنند. مکانیسم‌ها متفاوت هستند، اما هر دو اساساً راه‌هایی هستند که یک client می‌تواند بخشی از داده‌ها را که می‌خواهد در مورد آن‌ها بداند، انتخاب کند.
  • هنگام query کردن یک پایگاه داده، نتیجه معمولاً بر اساس یک snapshot نقطه‌ای از داده‌ها است؛ اگر یک client دیگر متعاقباً چیزی را در پایگاه داده بنویسد که نتیجه query را تغییر دهد، client اول نمی‌فهمد که نتیجه قبلی‌اش اکنون منسوخ شده است (مگر این‌که query را تکرار کند، یا برای تغییرات polls کند). در مقابل، message brokers از queries دلخواه پشتیبانی نمی‌کنند، اما به clients هنگام تغییر داده‌ها (یعنی، زمانی که پیام‌های جدید در دسترس قرار می‌گیرند) اطلاع می‌دهند.

این دیدگاه سنتی از message brokers است، که در استانداردهایی مانند JMS [14] و AMQP [15] محصور شده است و در نرم‌افزارهایی مانند RabbitMQ، ActiveMQ، HornetQ، Qpid، TIBCO Enterprise Message Service، IBM MQ، Azure Service Bus و Google Cloud Pub/Sub [16] پیاده‌سازی شده است.
