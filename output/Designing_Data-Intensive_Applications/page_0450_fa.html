بر روی یک اکوسیستم بزرگ از کتابخانه‌های موجود برای انجام کارهایی مانند parsing، تجزیه و تحلیل زبان طبیعی، تجزیه و تحلیل تصویر، و اجرای الگوریتم‌های عددی یا آماری است.

آزادی برای اجرای آسان کد دلخواه همان چیزی است که از دیرباز سیستم‌های پردازش batch با میراث MapReduce را از پایگاه‌های داده‌ی MPP متمایز کرده است (نگاه کنید به "مقایسه Hadoop با پایگاه‌های داده‌ی توزیع شده" در صفحه 414)؛ اگرچه پایگاه‌های داده دارای امکاناتی برای نوشتن توابع تعریف شده توسط کاربر هستند، اما اغلب استفاده از آن‌ها دست و پا گیر است و با مدیران بسته و سیستم‌های مدیریت وابستگی که در بیشتر زبان‌های برنامه‌نویسی به‌طور گسترده مورد استفاده قرار می‌گیرند (مانند Maven برای Java، npm برای JavaScript و Rubygems برای Ruby) به خوبی یکپارچه نشده‌اند.

با این حال، موتورهای dataflow دریافته‌اند که مزایایی نیز برای گنجاندن ویژگی‌های اعلانی بیشتر در زمینه‌هایی فراتر از joins وجود دارد. به عنوان مثال، اگر یک تابع callback فقط شامل یک شرط فیلتر ساده باشد، یا فقط چند field را از یک رکورد انتخاب کند، سپس در فراخوانی تابع برای هر رکورد، overhead CPU قابل توجهی وجود دارد. اگر چنین عملیات‌های فیلتر کردن و mapping ساده‌ای به روشی اعلانی بیان شوند، بهینه‌ساز query می‌تواند از layouts ذخیره‌سازی مبتنی بر column (نگاه کنید به "ذخیره‌سازی Column-Oriented" در صفحه 95) استفاده کند و فقط ستون‌های مورد نیاز را از دیسک بخواند. Hive، Spark DataFrames و Impala از اجرای vectorized استفاده می‌کنند (نگاه کنید به "پهنای باند حافظه و پردازش vectorized" در صفحه 99): تکرار داده‌ها در یک حلقه درونی تنگ که برای CPU caches مناسب است، و اجتناب از فراخوانی توابع. Spark bytecode JVM را تولید می‌کند [79] و Impala از LLVM برای تولید کد native برای این حلقه‌های درونی استفاده می‌کند [41].

با گنجاندن جنبه‌های اعلانی در APIs سطح بالای خود، و داشتن بهینه‌سازهای query که می‌توانند در طول اجرا از آن‌ها استفاده کنند، فریم‌ورک‌های پردازش batch شروع به شباهت بیشتر به پایگاه‌های داده‌ی MPP می‌کنند (و می‌توانند عملکرد قابل مقایسه‌ای را به دست آورند). در همان زمان، با داشتن قابلیت توسعه‌پذیری برای اجرای کد دلخواه و خواندن داده‌ها در فرمت‌های دلخواه، آن‌ها مزیت انعطاف‌پذیری خود را حفظ می‌کنند.

تخصص برای حوزه‌های مختلف

در حالی که قابلیت توسعه‌پذیری برای اجرای کد دلخواه مفید است، موارد رایج زیادی نیز وجود دارد که در آن‌ها الگوهای پردازش استاندارد دوباره تکرار می‌شوند، و بنابراین ارزش دارد که پیاده‌سازی‌های قابل استفاده مجدد از building blocks مشترک داشته باشید. به‌طور سنتی، پایگاه‌های داده‌ی MPP نیازهای تحلیلگران هوش تجاری و گزارش‌دهی کسب‌وکار را برآورده کرده‌اند، اما این تنها یکی از حوزه‌هایی است که در آن‌ها از پردازش batch استفاده می‌شود.

یک حوزه دیگر با اهمیت فزاینده، الگوریتم‌های آماری و عددی است که برای برنامه‌های machine learning مانند طبقه‌بندی و سیستم‌های توصیه مورد نیاز است. پیاده‌سازی‌های قابل استفاده مجدد در حال ظهور هستند: به عنوان مثال، Mahout الگوریتم‌های مختلفی را برای یادگیری ماشین بر روی MapReduce، Spark و Flink پیاده‌سازی می‌کند، در حالی که MADlib عملکرد مشابهی را در داخل یک پایگاه داده‌ی MPP رابطه‌ای (Apache HAWQ) پیاده‌سازی می‌کند [54]. 428 | فصل 10: پردازش Batch
