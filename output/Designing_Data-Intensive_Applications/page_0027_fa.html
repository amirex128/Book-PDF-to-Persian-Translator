شکل 1-1. یک معماری احتمالی برای یک سیستم data که چندین مؤلفه را ترکیب می‌کند.

هنگامی‌که چندین ابزار را برای ارائه یک service ترکیب می‌کنید، رابط service یا application programming interface (API) معمولاً این جزئیات پیاده‌سازی را از clients پنهان می‌کند. اکنون شما اساساً یک سیستم data جدید و با هدف خاص از اجزای کوچک‌تر و با هدف عمومی ایجاد کرده‌اید. سیستم data ترکیبی شما ممکن است تضمین‌های خاصی را ارائه دهد: به‌عنوان‌مثال، این‌که cache به‌درستی با نوشتن‌ها باطل یا به‌روزرسانی می‌شود تا clients خارجی نتایج سازگار را ببینند. شما اکنون نه تنها یک application developer هستید، بلکه یک طراح سیستم data نیز هستید.

اگر در حال طراحی یک سیستم یا service data هستید، سؤالات دشوار زیادی مطرح می‌شود. چگونه اطمینان حاصل می‌کنید که data صحیح و کامل باقی می‌ماند، حتی زمانی که در داخل مشکلی پیش می‌آید؟ چگونه عملکرد خوب و مداومی را به clients ارائه می‌دهید، حتی زمانی که بخش‌هایی از سیستم شما خراب شده‌اند؟ چگونه مقیاس می‌دهید تا افزایش بار را مدیریت کنید؟ یک API خوب برای service چگونه است؟

عوامل زیادی وجود دارد که ممکن است بر طراحی یک سیستم data تأثیر بگذارند، از جمله مهارت‌ها و تجربه افراد درگیر، وابستگی‌های سیستم‌های قدیمی، مقیاس زمانی برای تحویل، تحمل سازمان شما نسبت به انواع مختلف ریسک، محدودیت‌های نظارتی و غیره. این عوامل تا حد زیادی به شرایط بستگی دارد.
