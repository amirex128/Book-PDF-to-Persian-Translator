Linearizability

در یک database eventually consistent، اگر شما از دو replicas مختلف یک سوال یکسان را در یک زمان یکسان بپرسید، ممکن است دو پاسخ متفاوت دریافت کنید. این گیج‌کننده است. آیا خیلی ساده‌تر نبود اگر database می‌توانست این توهم را ایجاد کند که فقط یک replica وجود دارد (یعنی، فقط یک کپی از داده‌ها)؟ سپس هر client دیدگاه یکسانی از داده‌ها را خواهد داشت، و شما نیازی به نگرانی در مورد replication lag نخواهید داشت.

این ایده پشت linearizability [6] است (همچنین به عنوان atomic consistency [7]، strong consistency, immediate consistency، یا external consistency [8] شناخته می‌شود). تعریف دقیق linearizability، کاملاً ظریف است، و ما آن را در بقیه این بخش بررسی خواهیم کرد. اما ایده اساسی این است که یک system را طوری جلوه دهیم که گویی فقط یک کپی از داده‌ها وجود دارد، و تمام عملیات روی آن atomic هستند. با این guarantee، حتی اگر در واقعیت چندین replica وجود داشته باشد، application نیازی به نگرانی در مورد آنها ندارد.

در یک system linearizable، به محض اینکه یک client، یک write را با موفقیت تکمیل کرد، تمام clients ها که از database می‌خوانند، باید قادر به دیدن value که تازه نوشته شده است، باشند. حفظ توهم یک کپی واحد از داده‌ها به معنای تضمین این است که value خوانده شده، جدیدترین و به‌روزترین value است، و از یک cache یا replica stale نمی‌آید. به عبارت دیگر، linearizability یک recency guarantee است. برای روشن شدن این ایده، بیایید به یک مثال از یک system که linearizable نیست، نگاهی بیندازیم.

این system linearizable نیست، و باعث می‌شود طرفداران فوتبال گیج شوند.
