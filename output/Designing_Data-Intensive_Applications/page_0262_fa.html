پیاده‌سازی snapshot isolation با استفاده از multi-version objects.

هر row در یک table دارای یک فیلد created_by است که حاوی ID transaction است که این row را در table درج کرده است. علاوه بر این، هر row دارای یک فیلد deleted_by است که در ابتدا خالی است. اگر یک transaction یک row را حذف کند، row در واقع از database حذف نمی‌شود، اما با تنظیم فیلد deleted_by روی ID transaction که درخواست حذف را داده است، برای حذف علامت‌گذاری می‌شود. در زمانی دیگر، هنگامی که مشخص شد که هیچ transaction دیگری نمی‌تواند به داده‌های حذف شده دسترسی داشته باشد، یک فرآیند garbage collection در database هر row را که برای حذف علامت‌گذاری شده است حذف می‌کند و فضای آنها را آزاد می‌کند.

یک update در داخل به یک delete و یک create ترجمه می‌شود. به عنوان مثال، در شکل 7-7، transaction 13 مبلغ 100 دلار از حساب 2 کم می‌کند و balance را از 500 دلار به 400 دلار تغییر می‌دهد. table accounts در حال حاضر در واقع شامل دو rows برای حساب 2 است: یک row با موجودی 500 دلار که توسط transaction 13 علامت‌گذاری شده است، و یک row با موجودی 400 دلار که توسط transaction 13 ایجاد شده است.
