چالش‌ها

از نظر سهولت استفاده، از نظر تاریخی شما نمی‌توانستید کد سمت client-side را برای REST over HTTP application protocol خود مانند پیاده‌سازی‌های RPC تولید کنید. این اغلب باعث شده است که افراد REST APIs را ایجاد کنند که client libraries را برای استفاده مصرف‌کنندگان ارائه می‌دهند. این client libraries یک binding به API به شما می‌دهند تا client integration را آسان‌تر کنید. مشکل این است که client libraries می‌توانند باعث ایجاد برخی چالش‌ها در رابطه با coupling بین client و server شوند، چیزی که ما در "DRY and the Perils of Code Reuse in a Microservice World" در صفحه 154 در مورد آن بحث خواهیم کرد.

در سال‌های اخیر این مشکل تا حدودی برطرف شده است. OpenAPI specification که از پروژه Swagger رشد کرد، اکنون این امکان را به شما می‌دهد که اطلاعات کافی را در یک REST endpoint تعریف کنید تا امکان تولید کد سمت client-side را در انواع زبان‌ها فراهم کنید. در تجربه من، من تیم‌های زیادی را ندیده‌ام که واقعاً از این قابلیت استفاده می‌کنند، حتی اگر قبلاً از Swagger برای documentation استفاده می‌کردند. من این گمان را دارم که این ممکن است به دلیل دشواری‌های اصلاح استفاده از آن در APIs فعلی باشد. من همچنین نگرانی‌هایی در مورد specification دارم که قبلاً فقط برای documentation استفاده می‌شد و اکنون برای تعریف یک قرارداد صریح‌تر استفاده می‌شود. این می‌تواند منجر به یک specification بسیار پیچیده‌تر شود—مقایسه یک OpenAPI schema با یک protocol buffer schema، به عنوان مثال، یک تضاد آشکار است. با وجود تحفظات من، خوب است که این گزینه اکنون وجود دارد.

عملکرد نیز ممکن است یک مسئله باشد. REST over HTTP payloads در واقع می‌تواند فشرده‌تر از SOAP باشد زیرا REST از formats جایگزین مانند JSON یا حتی binary پشتیبانی می‌کند، اما هنوز هم به اندازه یک binary protocol سبک مانند Thrift نخواهد بود. Overhead HTTP برای هر request نیز ممکن است برای الزامات low-latency نگران‌کننده باشد. همه HTTP protocols اصلی در حال حاضر از استفاده از Transmission Control Protocol (TCP) در زیر استفاده می‌کنند، که در مقایسه با سایر networking protocols ناکارآمدی دارد، و برخی از پیاده‌سازی‌های RPC به شما اجازه می‌دهند از networking protocols جایگزین به جای TCP مانند User Datagram Protocol (UDP) استفاده کنید.

محدودیت‌های اعمال شده بر HTTP به دلیل الزام به استفاده از TCP در حال رسیدگی است. HTTP/3، که در حال حاضر در حال نهایی شدن است، به دنبال تغییر به استفاده از QUIC protocol جدیدتر است. QUIC همان نوع قابلیت‌ها را به عنوان TCP (مانند بهبود تضمین‌ها نسبت به UDP) اما با برخی از پیشرفت‌های قابل توجه ارائه می‌دهد که نشان داده شده است بهبودهایی در latency و کاهش bandwidth را ارائه می‌دهند. احتمالاً چندین سال طول می‌کشد تا HTTP/3 تأثیر گسترده‌ای بر اینترنت عمومی داشته باشد، اما به نظر می‌رسد منطقی است که فرض کنیم سازمان‌ها می‌توانند زودتر از این در شبکه‌های خودشان بهره‌مند شوند.

با توجه به HATEOAS به طور خاص، می‌توانید با مسائل عملکردی بیشتری مواجه شوید. از آنجایی که clients نیاز به navigate کردن چندین control برای یافتن endpoints مناسب برای یک عملیات داده شده دارند، این می‌تواند منجر به protocols بسیار chatty شود—ممکن است چندین round trips وجود داشته باشد
