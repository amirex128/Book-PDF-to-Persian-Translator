Branching Models

تعدادی از موضوعات پیرامون build و deployment به نظر می‌رسد که باعث ایجاد اختلاف نظر زیادی در مورد استفاده از source code branching برای توسعه feature می‌شود. Branching در source code به شما امکان می‌دهد تا توسعه را به صورت مجزا انجام دهید، بدون اینکه کار دیگران را مختل کنید. در ظاهر، ایجاد یک source code branch برای هر feature که روی آن کار می‌شود—در غیر این صورت به عنوان feature branching شناخته می‌شود—یک مفهوم مفید به نظر می‌رسد.

مشکل این است که وقتی شما روی یک feature branch کار می‌کنید، شما به‌طور منظم تغییرات خود را با دیگران ادغام نمی‌کنید. اساساً، شما ادغام را به تأخیر می‌اندازید. و وقتی در نهایت تصمیم به ادغام تغییرات خود با دیگران می‌گیرید، یک merge بسیار پیچیده‌تری خواهید داشت.

رویکرد جایگزین این است که همه به همان "trunk" از source code وارد شوند. برای جلوگیری از تأثیر تغییرات بر افراد دیگر، از تکنیک‌هایی مانند feature flags برای "پنهان کردن" کار ناتمام استفاده می‌شود. این تکنیک که همه از یک trunk یکسان کار می‌کنند، توسعه مبتنی بر trunk نامیده می‌شود.

بحث در مورد این موضوع ظریف است، اما برداشت من این است که مزایای ادغام مکرر—و اعتبارسنجی آن ادغام—به اندازه‌ای قابل توجه است که توسعه مبتنی بر trunk، سبک توسعه ترجیحی من است. علاوه بر این، کار برای پیاده‌سازی feature flags، اغلب از نظر progressive delivery مفید است، مفهومی که ما در فصل 8 به آن خواهیم پرداخت.

Be Careful About Branches

زود ادغام کنید، و اغلب ادغام کنید. از استفاده از long-lived branches برای توسعه feature اجتناب کنید، و به جای آن توسعه مبتنی بر trunk را در نظر بگیرید. اگر واقعاً مجبور به استفاده از branches هستید، آن‌ها را کوتاه نگه دارید!
