بسیاری از اوقات، تنها چیزی که یک API gateway در واقع برای آن استفاده می‌شود، مدیریت دسترسی به microservices یک سازمان از GUI clients خود (صفحات وب، برنامه‌های native mobile) از طریق اینترنت عمومی است. در اینجا هیچ "third party"ای در این ترکیب وجود ندارد. نیاز به نوعی API gateway برای Kubernetes ضروری است، زیرا Kubernetes بومی فقط شبکه را در داخل cluster مدیریت می‌کند و هیچ کاری در مورد مدیریت ارتباط با و از خود cluster انجام نمی‌دهد. اما در چنین موردی استفاده، یک API gateway که برای دسترسی خارجی third-party طراحی شده است، بسیار زیاد است.

بنابراین اگر شما یک API gateway می‌خواهید، در مورد آنچه از آن انتظار دارید، واقعاً واضح باشید. در واقع، من کمی فراتر خواهم رفت و می‌گویم که شما احتمالاً باید از داشتن یک API gateway که بیش از حد انجام می‌دهد، اجتناب کنید. اما به این موضوع در ادامه خواهیم پرداخت.

Where to use them

هنگامی که شما شروع به درک این می‌کنید که چه نوع موارد استفاده‌ای دارید، دیدن اینکه به چه نوع gateway نیاز دارید، کمی آسان‌تر می‌شود. اگر فقط یک مورد از قرار دادن microservices در Kubernetes است، می‌توانید reverse proxies خود را اجرا کنید—یا بهتر از آن، می‌توانید به یک محصول متمرکز مانند Ambassador نگاه کنید، که از ابتدا با آن مورد استفاده در نظر گرفته شده است. اگر واقعاً متوجه شدید که نیاز به مدیریت تعداد زیادی از کاربران third-party دارید که به API شما دسترسی دارند، احتمالاً محصولات دیگری نیز وجود دارد که باید به آن‌ها نگاه کنید. در واقع، این امکان وجود دارد که شما ممکن است در نهایت بیش از یک gateway در این ترکیب داشته باشید تا جداسازی بهتر concerns را مدیریت کنید، و من می‌توانم ببینم که این در بسیاری از موقعیت‌ها منطقی است، اگرچه هشدار‌های معمول در مورد افزایش پیچیدگی کلی سیستم و افزایش network hops همچنان اعمال می‌شود.

من گاهی درگیر کار مستقیم با فروشندگان برای کمک به انتخاب ابزار بوده‌ام. من می‌توانم بدون هیچ تردیدی بگویم که در فضای API gateway، سوء فروش و رفتار ضعیف یا بی‌رحمانه‌ای را بیشتر از هر جای دیگری تجربه کرده‌ام—و در نتیجه شما ارجاعاتی به برخی از محصولات فروشنده در این فصل پیدا نخواهید کرد. من بیشتر این را به شرکت‌های تحت حمایت VC نسبت داده‌ام که محصولی را برای دوران رونق اقتصاد API ساخته‌اند، فقط برای اینکه متوجه شوند که آن بازار وجود ندارد، و بنابراین آن‌ها در دو جبهه می‌جنگند: آن‌ها بر سر تعداد کمی از کاربرانی که واقعاً به آنچه gateways پیچیده‌تر ارائه می‌دهند، نیاز دارند، می‌جنگند، در حالی که همچنین تجارت خود را به محصولات API gateway متمرکزتر که برای اکثریت قریب به اتفاق نیازهای ساده‌تر ساخته شده‌اند، از دست می‌دهند.

What to avoid

تا حدی به دلیل ناامیدی آشکار برخی از فروشندگان API gateway، انواع ادعاها برای آنچه این محصولات می‌توانند انجام دهند، مطرح شده است. این امر منجر به سوء استفاده‌های زیادی از این محصولات شده است، و به نوبه خود به بی‌اعتمادی تأسف‌بار از آنچه که اساساً یک مفهوم کاملاً ساده است، منجر شده است. دو نمونه کلیدی که من از سوء استفاده از API gateways دیده‌ام، برای تجمیع call و بازنویسی protocol است، اما من همچنین فشار بیشتری برای استفاده از API gateways برای تماس‌های درون perimeter (شرق-غرب) نیز دیده‌ام.
