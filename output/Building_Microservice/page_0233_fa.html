سناریویی که در آن چندین تیم همگی در حال اشتراک این giant build هستند. چه کسی مسئول است؟

مسلماً، این رویکرد نوعی از monorepo است. با این حال، در عمل، اکثر پیاده‌سازی‌های monorepo که من دیده‌ام، چندین builds را به قسمت‌های مختلف repo نگاشت می‌کنند، که این موضوع را به زودی با جزئیات بیشتری بررسی خواهیم کرد. بنابراین شما می‌توانید این الگوی یک repo که به یک single build نگاشت می‌شود را به عنوان بدترین شکل monorepo برای کسانی که می‌خواهند چندین microservices را به طور مستقل مستقر کنند، ببینید.

در عمل، من تقریباً هرگز این رویکرد را استفاده نمی‌کنم، مگر در مراحل اولیه پروژه‌ها. راستش را بخواهید، هر یک از دو رویکرد زیر به طور قابل‌توجهی ترجیح داده می‌شوند، بنابراین ما به جای آن روی آن‌ها تمرکز خواهیم کرد.

Pattern: One Repository per Microservice (aka Multirepo)

با الگوی یک repository در هر microservice (که معمولاً وقتی با الگوی monorepo مقایسه می‌شود، به عنوان الگوی multirepo نامیده می‌شود)، کد هر microservice در source code repository خود ذخیره می‌شود، همانطور که در شکل 7-6 می‌بینیم. این رویکرد منجر به یک نگاشت مستقیم بین تغییرات source code و CI builds می‌شود.

شکل 7-6. source code هر microservice در یک source code repository جداگانه ذخیره می‌شود

هر تغییری در User source code repository، build مربوطه را فعال می‌کند، و اگر آن قبول شود، من یک نسخه جدید از User microservice خود را برای deployment در دسترس خواهم داشت. داشتن یک repository جداگانه برای هر microservice همچنین به شما امکان می‌دهد مالکیت را بر اساس هر repository تغییر دهید، چیزی که اگر می‌خواهید یک مدل مالکیت قوی برای microservices خود در نظر بگیرید (که به زودی بیشتر در مورد آن توضیح خواهیم داد) منطقی است.

با این حال، ماهیت سرراست این الگو، برخی از چالش‌ها را ایجاد می‌کند. به طور خاص، توسعه‌دهندگان ممکن است خود را در حال کار با چندین repositories به طور همزمان بیابند، که به خصوص اگر آن‌ها در تلاش برای ایجاد تغییراتی در چندین
