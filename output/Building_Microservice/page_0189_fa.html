Service meshes و API gateways به طور بالقوه می‌توانند به microservices اجازه دهند که کد را به اشتراک بگذارند بدون اینکه نیازی به ایجاد client libraries جدید یا microservices جدید داشته باشند. به زبان ساده (بسیار)، service meshes و API gateways می‌توانند به عنوان proxies بین microservices عمل کنند. این می‌تواند به این معنی باشد که آن‌ها ممکن است برای پیاده‌سازی برخی از رفتار‌های microservice-agnostic که در غیر این صورت باید در کد انجام می‌شد، مانند service discovery یا logging، استفاده شوند.

اگر شما از یک API gateway یا یک service mesh برای پیاده‌سازی رفتار مشترک و رایج برای microservices خود استفاده می‌کنید، ضروری است که این رفتار کاملاً عمومی باشد—به عبارت دیگر، اینکه رفتار در proxy هیچ ارتباطی با هیچ رفتار خاصی از یک microservice جداگانه نداشته باشد.

اکنون، پس از توضیح این، من همچنین باید توضیح دهم که جهان همیشه به این روشنی نیست. تعدادی از API gateways سعی می‌کنند قابلیت‌هایی را برای ترافیک شرق-غرب نیز ارائه دهند، اما این چیزی است که ما به زودی در مورد آن بحث خواهیم کرد. اول، بیایید به API gateways و انواع چیزهایی که می‌توانند انجام دهند، نگاهی بیندازیم.

API Gateways

با تمرکز بیشتر بر روی ترافیک شمال-جنوب، نگرانی اصلی API gateway در یک محیط microservices، نگاشت درخواست‌ها از طرف‌های خارجی به microservices داخلی است. این مسئولیت شبیه به چیزی است که شما می‌توانید با یک HTTP proxy ساده به آن برسید، و در واقع API gateways معمولاً ویژگی‌های بیشتری را بر روی محصولات HTTP proxy موجود ایجاد می‌کنند، و آن‌ها تا حد زیادی به عنوان reverse proxies عمل می‌کنند. علاوه بر این، API gateways می‌توانند برای پیاده‌سازی مکانیسم‌هایی مانند API keys برای طرف‌های خارجی، logging، rate limiting و موارد مشابه استفاده شوند. برخی از محصولات API gateway، پورتال‌های توسعه‌دهنده را نیز ارائه می‌دهند، که اغلب مخاطب مصرف‌کنندگان خارجی هستند.

بخشی از سردرگمی در مورد API gateway مربوط به تاریخچه است. مدتی پیش، مقدار زیادی علاقه به چیزی وجود داشت که به آن "اقتصاد API" می‌گفتند. صنعت شروع به درک قدرت ارائه APIs به راه‌حل‌های مدیریت شده کرده بود، از محصولات SaaS مانند Salesforce تا پلتفرم‌هایی مانند AWS، زیرا مشخص بود که یک API به مشتریان انعطاف‌پذیری بیشتری در نحوه استفاده از نرم‌افزارشان می‌دهد. این امر باعث شد که تعدادی از افراد شروع به بررسی نرم‌افزارهایی کنند که قبلاً داشتند و مزایای قرار دادن آن عملکرد در معرض دید مشتریان خود، نه فقط از طریق یک GUI، بلکه از طریق یک API را در نظر بگیرند. امید این بود که این امر فرصت‌های بزرگ‌تر بازار را باز کند، و خوب، پول بیشتری به دست آورد. در میان این علاقه، تعدادی از محصولات API gateway پدیدار شدند تا به تحقق این اهداف کمک کنند. مجموعه ویژگی‌های آن‌ها به شدت به مدیریت API keys برای اشخاص ثالث، اعمال محدودیت‌های نرخ، و ردیابی استفاده برای مقاصد charge-back متمایل بود. واقعیت این است که در حالی که APIs قطعاً به عنوان یک راه عالی برای ارائه services به برخی از مشتریان نشان داده شده‌اند، اندازه اقتصاد API به اندازه آنچه برخی امیدوار بودند بزرگ نبود، و بسیاری از شرکت‌ها دریافتند که محصولات API gateway را خریداری کرده‌اند که مملو از ویژگی‌هایی است که هرگز واقعاً به آن‌ها نیاز نداشتند.
