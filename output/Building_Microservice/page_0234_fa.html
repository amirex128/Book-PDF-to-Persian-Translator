Reusing code across repositories

هنگام استفاده از این الگو، هیچ چیز مانع از این نمی‌شود که یک microservice به کد دیگری که در repositories مختلف مدیریت می‌شود، وابسته باشد. یک مکانیسم ساده برای انجام این کار این است که کدی را که می‌خواهید دوباره استفاده کنید، در یک library بسته‌بندی کنید که سپس به یک وابستگی صریح microservices بالادستی تبدیل می‌شود. ما می‌توانیم نمونه‌ای از آن را در شکل 7-7 ببینیم، که در آن Invoice و Payroll services هر دو از Connection library استفاده می‌کنند.

شکل 7-7. استفاده مجدد از کد در repositories مختلف

اگر شما می‌خواستید تغییری را در Connection library اعمال کنید، شما باید تغییراتی را در source code repository مربوطه ایجاد کنید و منتظر بمانید تا build آن کامل شود، و یک artifact با نسخه جدید به شما بدهد. برای استقرار نسخه‌های جدید از Invoice یا Payroll services با استفاده از این نسخه جدید از library، شما باید نسخه Connection library را که استفاده می‌کنند، تغییر دهید. این ممکن است نیاز به یک تغییر دستی داشته باشد (اگر شما به یک نسخه خاص وابسته هستید)، یا می‌تواند برای این اتفاق به صورت پویا پیکربندی شود، که بستگی به ماهیت CI tooling دارد که شما استفاده می‌کنید. مفاهیم پشت این موضوع با جزئیات بیشتری در کتاب Continuous Delivery توسط Jez Humble و Dave Farley آمده است.4
