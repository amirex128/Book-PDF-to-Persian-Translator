Client libraries

من با بیش از یک تیم صحبت کرده‌ام که اصرار داشتند که ایجاد client libraries برای services شما، یک بخش ضروری از ایجاد services در وهله اول است. استدلال این است که این امر استفاده از service شما را آسان می‌کند و از تکرار کد مورد نیاز برای استفاده از خود service جلوگیری می‌کند.

البته، مشکل این است که اگر همان افراد هم server API و هم client API را ایجاد کنند، این خطر وجود دارد که منطقی که باید در server وجود داشته باشد، شروع به نشت به client کند. من باید بدانم: من خودم این کار را انجام داده‌ام. هرچه منطق بیشتری وارد client library شود، هم‌افزایی شروع به فروپاشی می‌کند، و شما خود را مجبور می‌کنید که چندین clients را تغییر دهید تا fixes را در server خود اعمال کنید. شما همچنین انتخاب‌های فناوری را محدود می‌کنید، به خصوص اگر شما این را الزامی کنید که client library باید استفاده شود.

یک مدل که من برای client libraries دوست دارم، مدل Amazon Web Services (AWS) است. تماس‌های underlying SOAP یا REST web service را می‌توان مستقیماً برقرار کرد، اما همه فقط از یکی از software development kits (SDKs) موجود استفاده می‌کنند، که abstractions را بر روی underlying API ارائه می‌دهند. با این حال، این SDKs توسط جامعه گسترده‌تر، یا توسط افرادی در داخل AWS غیر از کسانی که روی خود API کار می‌کنند، نوشته شده‌اند. این درجه از جداسازی به نظر می‌رسد که کارآمد است و از برخی از مشکلات client libraries جلوگیری می‌کند. بخشی از دلیل اینکه این کار اینقدر خوب جواب می‌دهد این است که client مسئول زمان وقوع ارتقا است. اگر خودتان در مسیر client libraries قدم می‌گذارید، مطمئن شوید که این مورد وجود دارد.

Netflix به ویژه تأکید ویژه‌ای بر client library دارد، اما من نگرانم که مردم این را صرفاً از طریق لنز اجتناب از تکرار کد مشاهده کنند. در واقع، client libraries مورد استفاده توسط Netflix، اگر بیشتر نباشد، بیشتر در مورد اطمینان از قابلیت اطمینان و مقیاس‌پذیری سیستم‌های آن‌ها است. Netflix client libraries، service discovery، failure modes، logging، و سایر جنبه‌هایی را مدیریت می‌کنند که در واقع در مورد ماهیت خود service نیستند. بدون این shared clients، اطمینان از اینکه هر بخش از ارتباطات client/server در مقیاس وسیعی که Netflix کار می‌کند، به خوبی رفتار می‌کند، دشوار خواهد بود. استفاده آن‌ها در Netflix قطعاً راه‌اندازی و افزایش بهره‌وری را آسان کرده است و در عین حال اطمینان حاصل کرده است که سیستم به خوبی رفتار می‌کند. با این حال، طبق گفته حداقل یک نفر در Netflix، با گذشت زمان این امر منجر به درجه‌ای از coupling بین client و server شده است که مشکل‌ساز بوده است.

اگر رویکرد client library چیزی است که شما به آن فکر می‌کنید، مهم است که کد client را جدا کنید تا transport protocol اساسی را مدیریت کند، که می‌تواند با مواردی مانند service discovery و failure سروکار داشته باشد، از مواردی که مربوط به خود destination service هستند. تصمیم بگیرید که آیا می‌خواهید اصرار داشته باشید که از client library استفاده شود، یا اگر به افراد اجازه دهید از technology stacks مختلف برای برقراری calls به underlying API استفاده کنند. و در نهایت، اطمینان حاصل کنید که clients مسئول زمان ارتقاء client libraries خود هستند: ما باید اطمینان حاصل کنیم که توانایی انتشار services خود را مستقل از یکدیگر حفظ می‌کنیم!
