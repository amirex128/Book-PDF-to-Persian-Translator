هر دو از same underlying technology استفاده می‌کنند اما از نیاز به یک explicit service definition اجتناب می‌کنند، زیرا service definition به طور ضمنی توسط Java type definitions ارائه می‌شود.

با این حال، تمام این فناوری‌ها دارای همان core characteristic هستند: آن‌ها یک remote call را شبیه یک local call جلوه می‌دهند.

معمولاً، استفاده از یک فناوری RPC به این معنی است که شما در حال خرید یک serialization protocol هستید. RPC framework نحوه serialize و deserialize کردن داده‌ها را تعریف می‌کند. به عنوان مثال، gRPC از protocol buffer serialization format برای این منظور استفاده می‌کند. برخی از پیاده‌سازی‌ها به یک specific networking protocol (مانند SOAP، که استفاده کمی از HTTP می‌کند) متصل هستند، در حالی که برخی دیگر ممکن است به شما اجازه دهند از انواع مختلف networking protocols استفاده کنید، که می‌تواند ویژگی‌های اضافی را ارائه دهد. به عنوان مثال، TCP تضمین‌هایی در مورد تحویل ارائه می‌دهد، در حالی که UDP این کار را نمی‌کند اما دارای overhead بسیار کمتری است. این می‌تواند به شما اجازه دهد از networking technology متفاوتی برای موارد استفاده مختلف استفاده کنید.

RPC frameworks که دارای یک explicit schema هستند، ایجاد client code را بسیار آسان می‌کنند. این می‌تواند از نیاز به client libraries جلوگیری کند، زیرا هر client می‌تواند کد خود را در برابر این service specification تولید کند. با این حال، برای اینکه client-side code generation کار کند، client به روشی برای دریافت schema out of band نیاز دارد—به عبارت دیگر، consumer باید قبل از اینکه قصد داشته باشد calls برقرار کند، به schema دسترسی داشته باشد. Avro RPC یک outlier جالب در اینجا است، زیرا این گزینه را دارد که full schema را همراه با payload ارسال کند و به clients اجازه می‌دهد تا schema را به صورت پویا تفسیر کنند.

سهولت تولید client-side code یکی از نقاط فروش اصلی RPC است. این واقعیت که من فقط می‌توانم یک normal method call برقرار کنم و از بقیه صرف نظر کنم، یک مزیت بزرگ است.

چالش‌ها

همانطور که دیدیم، RPC مزایای زیادی را ارائه می‌دهد، اما خالی از اشکالات نیست—و برخی از پیاده‌سازی‌های RPC می‌توانند مشکل‌سازتر از بقیه باشند. بسیاری از این مسائل را می‌توان حل کرد، اما سزاوار بررسی بیشتر هستند.

  • Technology coupling.

    • برخی از RPC mechanisms، مانند Java RMI، به شدت به یک specific platform متصل هستند، که می‌تواند محدود کند که چه فناوری می‌تواند در client و server استفاده شود. Thrift و gRPC دارای مقدار چشمگیری از پشتیبانی برای زبان‌های جایگزین هستند، که می‌تواند این نقطه ضعف را تا حدودی کاهش دهد، اما آگاه باشید که فناوری RPC گاهی اوقات با محدودیت‌هایی در interoperability همراه است.

    به نوعی، این technology coupling می‌تواند نوعی از افشای جزئیات پیاده‌سازی فنی داخلی باشد. به عنوان مثال، استفاده از RMI نه تنها client را به JVM متصل می‌کند، بلکه server را نیز متصل می‌کند.

صادقانه بگویم، تعدادی از پیاده‌سازی‌های RPC وجود دارد که این محدودیت را ندارند—gRPC, SOAP و Thrift همه نمونه‌هایی هستند که interoperability بین different technology stacks را امکان‌پذیر می‌کنند.
