شکل 5-2. Topics به چندین مشترک اجازه می‌دهند تا messages یکسانی را دریافت کنند، که برای پخش event مفید است

Guaranteed delivery

پس چرا از یک broker استفاده کنیم؟ اساساً، آن‌ها برخی از قابلیت‌ها را ارائه می‌دهند که می‌تواند برای ارتباط asynchronous بسیار مفید باشد. ویژگی‌هایی که آن‌ها ارائه می‌دهند متفاوت است، اما جالب‌ترین ویژگی، guaranteed delivery است، چیزی که همه brokers پرکاربرد به نوعی از آن پشتیبانی می‌کنند. Guaranteed delivery تعهدی از سوی broker را توصیف می‌کند تا اطمینان حاصل شود که message تحویل داده شده است.

از دیدگاه microservice که message را ارسال می‌کند، این می‌تواند بسیار مفید باشد. اگر destination مورد نظر در دسترس نباشد، مشکلی نیست—broker message را نگه می‌دارد تا زمانی که بتواند تحویل داده شود. این می‌تواند تعداد چیزهایی را که یک upstream microservice باید در مورد آن‌ها نگران باشد، کاهش دهد. آن را با یک synchronous direct call—به عنوان مثال، یک HTTP request مقایسه کنید: اگر destination مورد نظر قابل دسترسی نباشد، upstream microservice باید بداند که با request چه کند؛ آیا باید call را دوباره امتحان کند یا تسلیم شود؟

برای اینکه guaranteed delivery کار کند، یک broker باید اطمینان حاصل کند که هر messages که هنوز تحویل داده نشده‌اند، به روشی پایدار نگه داشته می‌شوند تا زمانی که بتوانند تحویل داده شوند. برای عمل به این وعده، یک broker معمولاً به عنوان نوعی سیستم مبتنی بر cluster اجرا می‌شود، و اطمینان حاصل می‌کند که از دست رفتن یک ماشین واحد باعث از دست رفتن یک message نمی‌شود.

به طور معمول، موارد زیادی در اجرای صحیح یک broker وجود دارد، که تا حدی به دلیل چالش‌ها در مدیریت نرم‌افزار مبتنی بر cluster است. اغلب، وعده guaranteed delivery
