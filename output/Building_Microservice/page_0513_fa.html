BFFs برای وب دسکتاپ و فراتر از آن (ادامه)

من حداقل یک سازمان را دیده‌ام که از BFFs برای سایر اشخاص ثالثی که نیاز به برقراری تماس دارند، استفاده می‌کند. با بازگشت به مثال همیشگی من از MusicCorp، من ممکن است یک BFF را در معرض دید قرار دهم تا به اشخاص ثالث اجازه دهم اطلاعات پرداخت حق امتیاز را استخراج کنند، یا به پخش جریانی به طیف وسیعی از دستگاه‌های set-top box اجازه دهم، همانطور که در شکل 14-16 می‌بینیم. اینها دیگر واقعاً BFFs نیستند، زیرا اشخاص ثالث "رابط کاربری" ارائه نمی‌دهند، اما این نمونه‌ای از همان الگو است که در یک زمینه متفاوت استفاده می‌شود، بنابراین فکر کردم ارزش به اشتراک‌گذاری دارد.

شکل 14-16. استفاده از BFFs برای مدیریت APIs خارجی

این رویکرد می‌تواند به‌ویژه مؤثر باشد، زیرا اشخاص ثالث اغلب توانایی محدودی (یا تمایلی) به استفاده یا تغییر تماس‌های API که برقرار می‌کنند ندارند. با یک backend API مرکزی، ممکن است مجبور شوید نسخه‌های قدیمی API را فقط برای راضی کردن زیرمجموعه کوچکی از اشخاص خارجی خود که قادر به ایجاد تغییر نیستند، نگه دارید. با یک BFF، این مشکل به‌طور قابل توجهی کاهش می‌یابد. این امر همچنین تأثیر تغییرات را محدود می‌کند. شما می‌توانید API را برای فیس‌بوک به گونه‌ای تغییر دهید که با سایر اشخاص ناسازگار شود، اما از آنجایی که آنها از یک BFF متفاوت استفاده می‌کنند، تحت تأثیر این تغییر قرار نمی‌گیرند.

چه زمانی از آن استفاده کنیم

برای برنامه‌ای که فقط یک UI وب را ارائه می‌دهد، من معتقدم که BFF فقط در صورتی منطقی است که شما مقدار قابل توجهی از تجمیع را در سمت سرور نیاز داشته باشید. در غیر این صورت، من فکر می‌کنم برخی از تکنیک‌های ترکیب UI دیگری که قبلاً پوشش داده‌ایم، می‌توانند به همان خوبی و بدون نیاز به یک مؤلفه سمت سرور اضافی کار کنند.
