مشکلاتی برای یک database ایجاد می‌کند و به طور بالقوه تأثیر زیادی بر سیستم گسترده‌تر دارد. همان مشکل در مورد GraphQL نیز صدق می‌کند. تفاوت این است که با SQL ما حداقل ابزارهایی مانند query planners برای databases خود داریم، که می‌تواند به ما در تشخیص queries مشکل‌ساز کمک کند، در حالی که یک مشکل مشابه با GraphQL ممکن است ردیابی آن دشوارتر باشد.

Server-side throttling requests یک راه‌حل بالقوه است، اما از آنجایی که اجرای call ممکن است در چندین microservices پخش شود، این کار بسیار ساده نیست. در مقایسه با REST-based HTTP APIs معمولی، caching نیز پیچیده‌تر است.

با REST-based APIs، من می‌توانم یکی از بسیاری از response headers را تنظیم کنم تا به دستگاه‌های سمت client-side، یا intermediate caches مانند content delivery networks (CDNs)، کمک کنم تا responses را cache کنند تا دیگر نیازی به درخواست آن‌ها نباشد. این به همان روش با GraphQL امکان‌پذیر نیست. توصیه‌هایی که در مورد این موضوع دیده‌ام، به نظر می‌رسد که حول مرتبط کردن یک ID با هر resource بازگشتی (و به یاد داشته باشید، یک GraphQL query می‌تواند شامل چندین resource باشد) می‌چرخد و سپس دستگاه client را ملزم به cache کردن request در برابر آن ID می‌کند. تا جایی که من می‌دانم، این امر استفاده از CDNs یا caching reverse proxies را بدون کار اضافی، فوق‌العاده دشوار می‌کند.

اگرچه من برخی از راه‌حل‌های خاص پیاده‌سازی را برای این مشکل دیده‌ام (مانند آن‌هایی که در پیاده‌سازی JavaScript Apollo یافت می‌شوند)، caching به نظر می‌رسد که یا آگاهانه یا ناخودآگاهانه به عنوان بخشی از توسعه اولیه GraphQL نادیده گرفته شده است.

البته، اگر queries که شما صادر می‌کنید، در طبیعت به یک user خاص بسیار خاص هستند، این کمبود request-level caching ممکن است یک عامل تعیین‌کننده نباشد، زیرا cache-hit ratio شما احتمالاً کم خواهد بود. با این حال، من تعجب می‌کنم که آیا این محدودیت به این معنی است که شما همچنان با یک راه‌حل ترکیبی برای دستگاه‌های client، با برخی از requests (بیشتر عمومی) که از طریق normal REST-based HTTP APIs و سایر requests که از طریق GraphQL می‌روند، به پایان خواهید رسید.

مسئله دیگری این است که در حالی که GraphQL از نظر تئوری می‌تواند writes را مدیریت کند، به نظر نمی‌رسد که به خوبی برای reads مناسب باشد. این منجر به موقعیت‌هایی می‌شود که در آن تیم‌ها از GraphQL برای read استفاده می‌کنند اما از REST برای writes استفاده می‌کنند.

آخرین مسئله چیزی است که ممکن است کاملاً ذهنی باشد، اما من هنوز فکر می‌کنم ارزش مطرح کردن را دارد. GraphQL باعث می‌شود احساس کنید که شما فقط با داده‌ها کار می‌کنید، که می‌تواند این ایده را تقویت کند که microservices که شما با آن‌ها صحبت می‌کنید، فقط wrappers بر روی databases هستند. در واقع، من چندین نفر را دیده‌ام که GraphQL را با OData مقایسه می‌کنند، فناوری‌ای که به عنوان یک API عمومی برای دسترسی به داده‌ها از databases طراحی شده است. همانطور که قبلاً به تفصیل در مورد آن بحث کردیم، ایده برخورد با microservices فقط به عنوان wrappers بر روی databases می‌تواند بسیار مشکل‌ساز باشد. Microservices عملکرد را از طریق networked interfaces در معرض نمایش قرار می‌دهند. بخشی از این عملکرد ممکن است نیاز داشته باشد یا منجر به قرار گرفتن داده‌ها شود، اما آن‌ها همچنان باید منطق و رفتار داخلی خود را داشته باشند. فقط به این دلیل که شما از GraphQL استفاده می‌کنید، به این فکر نکنید که microservices شما چیزی بیش از یک API بر روی یک database هستند—ضروری است که GraphQL API شما به underlying datastores microservices شما متصل نباشد.
