Where to use it

جایگاه sweet GraphQL برای استفاده در perimeter سیستم است و عملکرد را برای external clients در معرض نمایش قرار می‌دهد. این clients معمولاً GUIs هستند، و با توجه به محدودیت‌های آن‌ها از نظر توانایی محدود آن‌ها در ارائه داده‌ها به end user و ماهیت شبکه‌های تلفن همراه، یک انتخاب واضح برای دستگاه‌های تلفن همراه است. اما GraphQL همچنین برای external APIs استفاده شده است، که GitHub یکی از اولین پذیرندگان GraphQL است. اگر یک API external دارید که اغلب از external clients می‌خواهد چندین calls را برای دریافت اطلاعات مورد نیاز خود انجام دهند، GraphQL می‌تواند به کارآمدتر و دوستانه‌تر شدن API کمک کند.

اساساً، GraphQL یک مکانیسم call aggregation و filtering است، بنابراین در زمینه معماری microservice از آن برای تجمیع calls بر روی چندین downstream microservices استفاده می‌شود. به این ترتیب، چیزی نیست که جایگزین ارتباط عمومی microservice-to-microservice شود.

جایگزینی برای استفاده از GraphQL این است که یک الگوی جایگزین مانند الگوی backend for frontend (BFF) را در نظر بگیرید—ما به آن نگاه خواهیم کرد و آن را با GraphQL و سایر تکنیک‌های تجمیع در فصل 14 مقایسه خواهیم کرد.

Message Brokers

Message brokers واسطه‌هایی هستند که اغلب به آن‌ها middleware گفته می‌شود، که بین processes قرار می‌گیرند تا ارتباط بین آن‌ها را مدیریت کنند. آن‌ها یک انتخاب محبوب برای کمک به پیاده‌سازی ارتباط asynchronous بین microservices هستند، زیرا انواع قابلیت‌های قدرتمند را ارائه می‌دهند.

همانطور که قبلاً بحث کردیم، یک message یک مفهوم عمومی است که چیزی را که یک message broker ارسال می‌کند تعریف می‌کند. یک message می‌تواند حاوی یک request، یک response یا یک event باشد. به جای اینکه یک microservice مستقیماً با یک microservice دیگر ارتباط برقرار کند، microservice در عوض یک message را به یک message broker می‌دهد، با اطلاعاتی در مورد نحوه ارسال message.

Topics و queues

Brokers تمایل دارند که queues یا topics را ارائه دهند، یا هر دو را. Queues معمولاً point to point هستند. یک sender یک message را در یک queue قرار می‌دهد، و یک consumer از آن queue می‌خواند.

با یک سیستم مبتنی بر topic، چندین consumer می‌توانند در یک topic مشترک شوند، و هر consumer مشترک یک کپی از آن message را دریافت خواهد کرد.

یک consumer می‌تواند نشان دهنده یک یا چند microservices باشد—معمولاً به عنوان یک consumer group مدل‌سازی می‌شود. این زمانی مفید خواهد بود که شما چندین نمونه از یک microservice داشته باشید، و بخواهید هر یک از آن‌ها بتوانند یک message را دریافت کنند. در شکل 5-1، ما یک مثال را می‌بینیم که در آن Order Processor دارای سه نمونه مستقر است، که همگی به عنوان بخشی از یک consumer group هستند. هنگامی که یک message در queue قرار می‌گیرد، فقط یک عضو از consumer group آن message را دریافت خواهد کرد؛ این بدان معناست که queue
