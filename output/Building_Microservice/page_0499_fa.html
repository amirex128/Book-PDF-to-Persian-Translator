Web Components

در نگاه اول، Web Component Standard باید یک راه بدیهی برای پیاده‌سازی این widgets باشد. Web Component Standard توضیح می‌دهد که چگونه می‌توانید یک جزء UI ایجاد کنید که می‌تواند جنبه‌های HTML، CSS و JavaScript خود را sandbox کند. متأسفانه، به نظر می‌رسد که web components standard مدت زیادی طول کشیده است تا مستقر شود، و مدت زمان بیشتری طول کشید تا به درستی توسط مرورگرها پشتیبانی شود. به نظر می‌رسد که بسیاری از کارهای اولیه در اطراف آنها متوقف شده است، که بدیهی است بر پذیرش تأثیر گذاشته است. من هنوز با سازمانی روبرو نشده‌ام که از web components ارائه شده از microservices استفاده کند، به عنوان مثال.

با توجه به اینکه Web Component Standard در حال حاضر نسبتاً خوب پشتیبانی می‌شود، این امکان وجود دارد که ما آنها را به عنوان یک راه رایج برای پیاده‌سازی widgets sandboxed یا micro frontends بزرگتر در آینده ببینیم، اما پس از سال‌ها انتظار برای وقوع این اتفاق، من نفس خود را حبس نمی‌کنم.

چه زمانی از آن استفاده کنیم

این الگو، اگر می‌خواهید تیم‌های stream-aligned را end-to-end اتخاذ کنید، ضروری است، جایی که شما در تلاش برای دور شدن از یک معماری لایه‌ای هستید. من همچنین می‌توانم تصور کنم که در موقعیتی مفید باشد که شما می‌خواهید یک معماری لایه‌ای را حفظ کنید، اما عملکرد frontend اکنون آنقدر بزرگ است که به چندین تیم frontend اختصاصی نیاز است.

یک مشکل کلیدی با این رویکرد وجود دارد که مطمئن نیستم قابل حل باشد. گاهی اوقات قابلیت‌های ارائه شده توسط یک microservice به خوبی در یک widget یا یک صفحه جای نمی‌گیرد. مطمئناً، من ممکن است بخواهم توصیه‌ها را در یک box در یک صفحه وب‌سایت خود نشان دهم، اما اگر بخواهم توصیه‌های پویا را در جای دیگری ادغام کنم چه؟ به عنوان مثال، وقتی جستجو می‌کنم، می‌خواهم type ahead به طور خودکار توصیه‌های تازه را فعال کند. هرچه یک شکل تعامل متقاطع‌تر باشد، احتمال اینکه این مدل مناسب باشد، کمتر است، و احتمال اینکه ما دوباره به تماس‌های API برگردیم، بیشتر است.
