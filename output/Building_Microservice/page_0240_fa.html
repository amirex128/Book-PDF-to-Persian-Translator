مثال 7-1. نمونه هایی از نحوه مشخص کردن مالکیت در دایرکتوری های خاص در یک GitHub فایل CODEOWNERS

# In this example, @doctocat owns any files in the build/logs # directory at the root of the repository and any of its # subdirectories. /build/logs/ @doctocat

# In this example, @octocat owns any file in an apps directory # anywhere in your repository. apps/ @octocat # In this example, @doctocat owns any file in the /docs # directory in the root of your repository. /docs/ @doctocat

مفهوم مالکیت کد خود GitHub تضمین می کند که صاحبان کد برای فایل های منبع هستند هر زمان که یک درخواست pull برای فایل های مربوطه مطرح می شود، برای بررسی درخواست می شود. این می تواند مشکل با درخواست های pull بزرگتر، زیرا می توانید نیاز به تأیید از چندین بازبین، اما دلایل خوبی برای هدف قرار دادن درخواست های pull کوچکتر وجود دارد، در هر صورت.

ابزارها

monorepo خود Google عظیم است و مقدار قابل توجهی مهندسی می طلبد برای کار در مقیاس. مواردی مانند یک سیستم build مبتنی بر گراف را در نظر بگیرید که داشته است نسل های متعددی را طی کرد، یک linker شیء توزیع شده برای سرعت بخشیدن به ساخت زمان‌ها، plug-in ها برای IDE ها و ویرایشگرهای متن که می‌توانند به طور پویا فایل‌های وابستگی را در بررسی—این مقدار زیادی کار است. با رشد Google، به طور فزاینده ای به آن ضربه زد محدودیت هایی در استفاده از Perforce و در نهایت مجبور به ایجاد اختصاصی خود شد ابزار کنترل منبع به نام Piper. وقتی در سال 2007–2008 در این بخش از Google کار می کردم، بیش از صد نفر ابزارهای مختلف توسعه دهندگان را نگهداری می کردند، با بخش مهمی از این تلاش برای رسیدگی به پیامدهای monorepo رویکرد. این چیزی است که شما می توانید آن را توجیه کنید اگر ده ها هزار engi - neers، البته.
